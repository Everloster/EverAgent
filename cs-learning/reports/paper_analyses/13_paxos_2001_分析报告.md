## 🎯 一句话总结

> Paxos 是解决分布式系统**单值共识**问题的协议，通过 Prepare/Promise/Accept/Accepted 两阶段握手，在多数节点存活的异步网络中保证**安全性（不会产生两个不同的决定值）**。

---

## 🌍 背景与动机（WHY）

### 分布式系统面临的根本挑战

在分布式系统中，多个节点需要对某个值达成一致（共识），但面临：

- **节点可能崩溃**（Crash Failure）
- **消息可能延迟、丢失、重排**（异步网络）
- **消息不会被篡改**（不考虑拜占庭错误）

**共识问题**：让一组节点对同一个提案值达成一致，且一旦决定不可改变。

### 历史背景：为什么有"Made Simple"版本？

Lamport 1989 年用一个**希腊岛屿"Paxos"上的议会**类比写了原始论文 *The Part-Time Parliament*，故事性太强导致审稿人迷惑，被 TOCS 拒稿多年（直到 1998 年才发表）。

2001 年，Lamport 用白话文重写，直接说"这是一个分布式算法"，去掉所有议会类比，即本文。

### Prior Work 的问题

- **两阶段提交（2PC）**：协调者（Coordinator）单点故障会导致阻塞
- **视图戳复制（Viewstamped Replication）**：等价于 Paxos，但表达方式不同
- **FLP 不可能定理**（Fischer-Lynch-Paterson 1985）：异步网络中**不存在**能容忍哪怕一个节点崩溃的确定性共识算法

**Paxos 的突破**：FLP 说的是确定性算法，Paxos 是**概率性终止**的——在良好网络条件下**最终**会终止，但不保证有界时间内完成。

---

## 💡 核心贡献（WHAT）

1. **清晰定义共识三要求**：
   - **Safety（安全性）**：只有一个值被选定，且被选定的值必须是某个 Proposer 提议的
   - **Liveness（活性）**：如果足够多节点存活，最终会有值被选定
   - **Agreement（一致性）**：一旦值被选定，所有节点最终都会学到该值

2. **两阶段协议**：用最少的消息轮次解决单值共识

3. **Multi-Paxos**：将单值共识扩展为**日志（有序决策序列）**的共识，奠定状态机复制（State Machine Replication）基础

## 🔧 技术方法（HOW）

### 角色定义

Paxos 中节点可以身兼多个角色：

```
Proposer（提案者）：发起提案，试图让某个值被接受
Acceptor（接受者）：对提案投票，多数派决定最终值
Learner（学习者）：学习最终被选定的值
```

实际部署中，每个节点通常同时扮演三个角色。

### Phase 1：Prepare / Promise（准备阶段）

```
步骤 1a - Proposer 发送 Prepare(n)：
  - Proposer 选择一个全局唯一的提案编号 n（通常用 时间戳 + 节点ID）
  - 向所有 Acceptors 广播 Prepare(n)

步骤 1b - Acceptor 处理 Prepare(n)：
  if n > maxN（我见过的最大编号）:
      maxN = n
      返回 Promise(n, accepted_n, accepted_v)
      # accepted_n: 我之前 Accept 过的最大提案编号（没有则返回null）
      # accepted_v: 对应的值（没有则返回null）
      # 承诺：不再接受编号 < n 的提案
  else:
      忽略或返回 Nack
```

**关键约束**：Acceptor 承诺（Promise）后，不再接受编号更小的提案。

### Phase 2：Accept / Accepted（接受阶段）

```
步骤 2a - Proposer 发送 Accept(n, v)：
  前提：收到多数派（> N/2）的 Promise 响应

  选择值 v 的规则：
    if 所有 Promise 响应中 accepted_n 都为 null:
        v = Proposer 自己想提议的值（自由选择）
    else:
        v = 所有 Promise 中 accepted_n 最大的那个对应的 accepted_v
        # 必须延续之前可能已被选定的值！

  向所有 Acceptors 广播 Accept(n, v)

步骤 2b - Acceptor 处理 Accept(n, v)：
  if n >= maxN（未违反 Promise）:
      accepted_n = n
      accepted_v = v
      向所有 Learners 发送 Accepted(n, v)
  else:
      忽略
```

**Learner 学习**：当 Learner 收到多数派的 Accepted(n, v) 后，v 被选定（Chosen）。

### 完整流程示意

```
时间轴：

Proposer P1:  Prepare(1) ──────────→ Acceptors A1, A2, A3
                        ←────────── Promise(1, null, null) × 多数派

              Accept(1, "X") ──────→ Acceptors
                        ←────────── Accepted(1, "X") × 多数派

Learner L:   学到 "X" 被选定 ✅

─────────────────────────────────────────
并发情况（两个 Proposer 竞争）：

P1:  Prepare(1) → A1,A2,A3
P2:  Prepare(2) → A1,A2,A3  （抢先）

A1,A2,A3 → 都 Promise(2, ...) 给 P2（编号2更大）
A1,A2,A3 → 拒绝 P1 的 Prepare(1) 或 Promise(1, ...) 但后来不响应其 Accept

P2 得到多数派 Promise → Accept(2, v2) 成功
P1 的 Accept(1, v1) 被拒绝（因为 Acceptor 已 Promise 给更大编号2）

最终：v2 被选定
```

## 🔬 深层理解：为什么这样设计？

### Safety 保证的关键：值的延续性

最难理解的是 Phase 2a 中"**为什么 Proposer 必须延续之前 accepted 的值**"。

**思想实验**：

```
假设值 "X" 已经被多数派接受（但 Proposer P1 崩溃，Learner 还不知道）：
A1: accepted("X", n=5)
A2: accepted("X", n=5)
A3: 没来得及接受

新 Proposer P2 发起 Prepare(n=6)：
- A1 回复 Promise(6, 5, "X")
- A2 回复 Promise(6, 5, "X")
- A3 回复 Promise(6, null, null)

P2 看到：多数派响应中，最大的 accepted_n = 5，对应 "X"
→ P2 必须选 v = "X"（即使 P2 本来想提议 "Y"）

如果 P2 强行提议 "Y"：
- 可能导致 A1,A2 接受 "Y"
- 而之前 A1,A2 接受的 "X" 实际上已经"被选定"了
- → Safety 违反！同一个 slot 出现两个值
```

**结论**：Phase 1 的 Promise 不只是"我不接受旧提案"，还在于**侦测是否有值已经被选定**。如果有，必须延续。

### Liveness 问题：活锁（Dueling Proposers）

Paxos 存在理论上的活锁风险：

```
P1: Prepare(1) → Promise
P2: Prepare(2) → Promise（覆盖P1的Promise）
P1: Accept(1) → 被拒（P2已Promise编号2）
P1: Prepare(3) → Promise（覆盖P2的Promise）
P2: Accept(2) → 被拒
P2: Prepare(4) → ...
```

两个 Proposer 无限互相覆盖，永远无法选定值。

**解决方案**：引入**唯一主 Proposer（Distinguished Leader）**——同一时刻只有一个 Proposer 活跃，其他 Proposer 等待。这就是 Multi-Paxos 的核心思想，也是 Raft 强 Leader 的来源。

## 📊 Multi-Paxos：从单值到日志

单值 Paxos 只能对**一个值**达成共识。实用系统需要对**一系列命令**（日志条目）达成共识，即 State Machine Replication（状态机复制）。

```
日志结构：
Slot 1: 对 v₁ 运行 Basic Paxos → 选定 cmd₁
Slot 2: 对 v₂ 运行 Basic Paxos → 选定 cmd₂
Slot 3: 对 v₃ 运行 Basic Paxos → 选定 cmd₃
...
```

**Multi-Paxos 优化**：选出稳定的 Leader 后，Phase 1（Prepare/Promise）只需做**一次**，后续所有 Slot 直接跳到 Phase 2（Accept），大幅减少消息数量：

```
普通 Paxos（每个 Slot）：2 轮消息 = Prepare + Accept
Multi-Paxos（稳定 Leader 后）：1 轮消息 = 仅 Accept
```

## 🏛️ 历史地位与影响

### Paxos 在工程界的遭遇

Paxos 虽然理论完美，但实现极其困难：

- **原论文过于理论化**，实现细节未指明
- **Multi-Paxos 到具体实现有巨大鸿沟**（leader 选举、日志压缩、成员变更等）
- Google 在 Chubby（分布式锁服务）实现 Paxos 后，专门写了一篇论文描述这有多难

这正是 **Raft（2014）** 出现的原因：Ongo 和 Ousterhout 明确以"可理解性"为目标，将 Multi-Paxos 的精髓重新组织为更清晰的三个独立子问题（Leader 选举、日志复制、安全性）。

### 工业界使用情况

| 系统 | 共识算法 |
|------|----------|
| Google Chubby | Paxos |
| Google Spanner | Multi-Paxos（每个 Paxos Group） |
| Apache Zookeeper | ZAB（Paxos 变体，更接近 Multi-Paxos） |
| etcd | Raft |
| CockroachDB | Raft |
| TiKV | Raft |

## 🤔 批判性视角

**Paxos 的问题**：
1. **活跃性不保证**：异步网络下可能永远无法选定值（活锁）
2. **Multi-Paxos 细节缺失**：Leader 选举、日志压缩、成员变更均未说明
3. **实现难度高**：Google 工程师花了数年才得到正确的 Chubby 实现
4. **消息数量多**：基本 Paxos 需要 2 轮往返（4 次消息延迟）

**Raft vs Paxos**：
- Raft 不是"更好的"算法，而是"更容易理解和实现"的协议
- Raft 的强 Leader 限制了写性能（所有写必须经 Leader），Paxos 理论上可以 Multi-Leader
- 但 Multi-Leader Paxos（如 EPaxos）实现复杂度极高

---

## 📌 关键收获（学习要点）

1. **共识三要求**：Safety（单一值）+ Liveness（最终决定）+ Agreement（所有人学到）
2. **两阶段核心**：Phase 1 侦测已有共识，Phase 2 提交并延续
3. **多数派是关键**：任意两个多数派必然有交集，保证信息传递
4. **值的延续性**：Proposer 必须延续见到的最大编号的已接受值，这是 Safety 的根本
5. **Paxos → Raft 的演进**：理解 Paxos 的难点，才能理解 Raft 的设计决策
