## TL;DR（一段话总结）

1987 年，Paul Mockapetllis 等人在 ARPA 的资助下发布了 RFC 1034（概念）和 RFC 1035（实现规范），定义了**域名系统（DNS）**——一个**分布式、层次化、可扩展的命名服务**，用于将人类可读的域名（如 `example.com`）映射为 IP 地址等网络资源。DNS 的核心创新是**将命名空间组织为倒置树形结构，通过委托授权（delegation）实现完全分布式管理**，并通过**缓存和查询类型分离（递归 vs 迭代）**在一致性和性能之间取得平衡。这一设计在 1987 年就被设计为"几乎不需要每次更新中央数据库"，39 年后的今天，全球约 3.5 亿个注册域名、数百亿台联网设备全部依赖 DNS 进行名称解析，其核心数据结构和查询协议几乎没有本质改变，证明了该设计的卓越远见。

## Step 1 | 背景与问题

### 历史节点

**互联网编号分配与命名管理的危机**：

| 年份 | 事件 | 意义 |
|------|------|------|
| **1970s** | ARPANET 使用 `/etc/hosts` 文件手动管理主机名到 IP 的映射 | 仅适用于数十台主机的规模 |
| **1981** | ARPANET 扩展到 hundreds of hosts，`hosts.txt` 文件通过 FTP 每周从 SRI-NIC 分发 | 文件越来越大，更新频率赶不上主机增长速度 |
| **1983** | ARPANET 全面切换到 TCP/IP（Flag Day） | 主机数量加速增长，`hosts.txt` 方法彻底不可行 |
| **1984** | 域名系统草案开始讨论，RFC 882/883（后被废弃）发布 | DNS 概念雏形 |
| **1987** | **RFC 1034/1035 正式发布** | DNS 成为官方标准 |
| **1988** | 第一个根服务器被部署，`.com`、`.edu`、`.gov` 等 TLD 开始运行 | DNS 正式进入生产环境 |
| **1990s-至今** | DNS 从 IP 解析扩展到 DNSSEC、TXT、MX、SRV 等记录类型 | 功能持续扩展，核心架构不变 |

**历史背景**：到 1984-1986 年，ARPANET 互联网社区面临一个紧迫的扩展危机：`hosts.txt` 文件每周从 SRI-NIC（一个中央机构）分发，但 ARPANET 主机数量已经从数十台增长到数千台，且 TCP/IP 的异构互联导致跨网络的主机命名问题更加复杂。同时，随着大学、企业、政府机构接入网络，**谁应该负责分配和管理名称**变成了一个组织难题——没有一个中央机构能够承担全球命名管理的职责。DNS 必须在技术性能和组织去中心化之间找到平衡。

### 核心问题

> **如何设计一个命名系统，能够以可扩展（Scalable）、分布式（Distributed）、容错（Fault-tolerant）的方式，将人类可读的域名映射为网络可理解的 IP 地址？**

这个问题分解为若干子问题：

1. **命名唯一性**：如何确保 `example.com` 只有一个合法所有者？
2. **查找效率**：给定一个域名，如何快速找到对应的 IP？
3. **分布式管理**：谁负责管理 `com` 下的域名？谁负责 `example.com`？
4. **动态更新**：IP 地址变化时，如何快速传播更新而不需要重载整个系统？
5. **可靠性**：解析器崩溃或网络分区时，名称解析能否继续工作？
6. **可扩展性**：系统能否支撑从数十台主机到数十亿设备的增长？

### 前人方案的不足

| 方案 | 核心问题 |
|------|----------|
| **HOSTS.TXT 集中文件**（1980 年代前） | 单点故障（SRI-NIC）、文件膨胀（每周更新时网络拥堵）、无层次结构、更新延迟高达数周 |
| **OSI X.500 目录服务** | 过于复杂（基于 X.500 协议簇），实现成本高，无法适应互联网的轻量级需求 |
| **本地 `/etc/hosts` 文件** | 无法跨网络同步；每台主机需要独立维护；无权威来源 |
| **Flat Namespace（平面命名）** | 名称必须全局唯一，无法聚合；在数十亿设备规模下不可行 |

### 论文的核心主张

> **一句话**：通过将命名空间组织为**倒置树形层次结构**，并允许**树中每个节点由不同机构独立管理**（委托授权），实现一个**无需中央数据库的全局命名服务**。

## Step 2 | 技术方案

### 核心思想

**直觉理解**：把 DNS 想象成**一个由无数子邮局组成的全球邮政系统**。

- 根邮局（Root Server）知道所有**国家**（TLD）邮局的地址，但不直接处理某个具体街道的邮件
- 国家邮局（如 `.com` 邮局）知道各个**公司**邮局的地址，但不直接处理某台个人电脑的邮件
- 公司邮局（如 `example.com`）知道各个**部门/服务器**的地址
- 最终，具体的收件地址被精确路由到目标服务器

当你向 `www.example.com` 发送请求时，你的本地 resolver 询问根服务器，根服务器告诉你"去找 `.com` 服务器"，`.com` 服务器告诉你"去找 `example.com` 授权服务器"，最后 `example.com` 服务器告诉你 `www` 的 IP 地址。

### 关键设计

#### 1. 层次化命名空间（Hierarchical Namespace）

DNS 的命名空间是一棵**倒置的树**（根节点在最顶层）：

```
                         [根节点] (.)
                         /   |   \
                       .com .org .net ... (TLD)
                        /       \
                example.com   google.com
                   /    \
            www.example.com  mail.example.com
```

**域（Domain）vs 主机（Host）**：
- **域**是一个子树及其所有子节点的名称（如 `example.com`）
- **主机**是树中的叶子节点，通常对应一台具体机器（如 `www.example.com`）

**命名规则**：
- 每个节点有一个**标签（Label）**，最多 63 个字符
- 从根到该节点的路径用`.`分隔，总长度不超过 255 个字符
- 不区分大小写
- 名称分为**完全限定域名（FQDN）**和**相对名称**

#### 2. 权威与委托授权（Authority & Delegation）

DNS 的最关键创新是**委托授权（Delegation）**——父 zone 可以将子 zone 的管理权移交给另一个 nameserver。

```
[根 zone (.       )] → 委托给 →
[TLD zone (.com   )] → 委托给 →
[example.com zone ] → 管理 →
[www.example.com  ]（A 记录：93.184.216.34）
```

**Zone 的概念**：
- Zone 是 DNS 树中**由同一权威机构管理**的一段连续子树
- Zone 和 domain 的区别：zone 包含 domain 但授权切分了管理边界
- 每个 zone 至少有 **2 个 nameserver**（冗余容错）

#### 3. 名字服务器（Nameserver）类型

| 类型 | 职责 | 举例 |
|------|------|------|
| **根服务器（Root Server）** | 知道所有 TLD 服务器的地址 | 全球 13 组根服务器（A-M root） |
| **TLD 服务器（TLD Server）** | 管理顶级域（如 .com、.org、.cn） | Verisign 运营 .com |
| **权威服务器（Authoritative Server）** | 持有 zone 的最终权威数据（SOA + 记录） | example.com 的 ns1.example.com |
| **递归服务器（Recursive Resolver）** | 替客户端完成完整查询路径，不持有数据 | ISP 的 DNS 服务器（8.8.8.8 等） |

#### 4. Resolver（解析器）

Resolver 是运行在**客户端机器上**的软件，负责：
1. 接收应用程序的查询请求（如"www.example.com 的 IP 是什么"）
2. 决定查询策略（使用缓存、使用递归还是迭代）
3. 与 authoritative nameserver 通信
4. 缓存收到的响应

**Resolver 的位置**：RFC 1034 将 resolver 定义为"client-side of DNS"，通常集成在操作系统的 DNS 客户端库（如 glibc 的 resolver，或 Windows DNS Client 服务）中。

#### 5. 查询类型：递归 vs 迭代

这是 DNS 设计的核心权衡：

**迭代查询（Iterative Query）**：
```
Client → Root Server   : "www.example.com?"
Root   → Client        : "我不知道，但我知道 .com server 的地址，去问它"
Client → .com Server   : "www.example.com?"
.com   → Client       : "我不知道，但我知道 example.com server 的地址，去问它"
Client → example.com   : "www.example.com?"
example.com → Client   : "93.184.216.34"
```
- 每一步返回的是**下一个应该查询的服务器地址**，而不是最终答案
- **节省 nameserver 资源**：服务器只返回自己知道的部分
- **客户端/Resolver 承担更多工作**

**递归查询（Recursive Query）**：
```
Client → Recursive Resolver : "www.example.com?"
Recursive Resolver 自行完成上述整个链条
Recursive Resolver → Client : "93.184.216.34"
```
- Recursive Resolver 承担完整查询工作
- **对客户端友好**：客户端只需发出一次请求
- **需要 resolver 有较强能力和较大缓存**

**RFC 1034 的设计选择**：允许 nameserver 同时支持递归和迭代查询，但鼓励在权威服务器上**禁用递归**（因为权威服务器的性能不应被递归查询消耗）；在递归服务器上**启用递归**（以提高缓存命中率）。

#### 6. 缓存策略（Caching）

DNS 缓存是性能的关键，但同时引入了**一致性**问题。

**缓存的来源**：
- Recursive Resolver 缓存所有中间响应（根、TLD、权威服务器的地址）
- **TTL（Time-to-Live）**：每条记录都有一个 TTL 值，告诉 resolver "这个答案可以缓存多久"
- 典型 TTL 值：知名网站通常 300-3600 秒（5分钟-1小时），权威记录可能 86400 秒（1天）

**Negative Caching（RFC 2308）**：
- 如果 `example.com` 的 nameserver 返回"不存在该子域名"（NXDOMAIN），这个否定答案本身也可以被缓存
- **RFC 1034 原版未定义 negative caching**，后来 RFC 2308（1998）补充了这一机制

**缓存一致性模型**：
- DNS 使用 **TTL 驱动的最终一致性**（Eventual Consistency）
- 当域名所有者更新 IP 地址时，旧数据可能继续被缓存（最多 TTL 秒）
- 这是 DNS 设计中对**性能和一致性**的权衡：**接受短暂的不一致性**以获得高读取性能

#### 7. 资源记录（Resource Records, RR）

DNS 的数据存储单元是一条**资源记录（RR）**：

```
[Name]  [TTL]  [CLASS]  [TYPE]  [RDATA]
```

| 字段 | 含义 | 示例 |
|------|------|------|
| **Name** | 域名 | `www.example.com` |
| **TTL** | 缓存时间（秒） | `3600` |
| **CLASS** | 协议族（IN = Internet，最常用） | `IN` |
| **TYPE** | 记录类型 | `A`、`AAAA`、`MX`、`TXT` |
| **RDATA** | 记录数据 | `93.184.216.34` |

**常见记录类型**：

| 类型 | 用途 | RDATA 示例 |
|------|------|------------|
| **A** | IPv4 地址映射 | `93.184.216.34` |
| **AAAA** | IPv6 地址映射 | `2606:2800:220:1::` |
| **CNAME** | 别名（canonical name） | `www.example.com → example.com` |
| **MX** | 邮件交换服务器 | `10 mail.example.com` |
| **TXT** | 文本信息（SPF、验证等） | `"v=spf1 include:_spf.example.com ~all"` |
| **NS** | 权威 nameserver | `ns1.example.com` |
| **SOA** | Start of Authority（zone 管理信息） | 序列号、主服务器、刷新间隔等 |

#### 8. Zone 数据结构与 AXFR/IXFR

Zone 数据在 nameserver 内部以**zone file**形式存储，包含：
- **SOA 记录**：zone 的管理信息（序列号、刷新间隔等）
- **NS 记录**：该 zone 的权威 nameserver
- **A/AAAA 记录**：主机地址
- **其他记录类型**

**Zone Transfer（区域传输）**：
- **AXFR（Full Zone Transfer）**：完整传输整个 zone 数据（TCP port 53）
- **IXFR（Incremental Zone Transfer）**：仅传输变更部分（RFC 1995）
- 用于 nameserver 之间的数据同步，是 **DNS 复制的核心机制**

### 权衡分析

| 获得 | 牺牲 |
|------|------|
| **完全分布式管理**（各机构独立管理子zone） | 跨 zone 的查询需要多次网络往返（延迟） |
| **高可扩展性**（树形层次天然支持水平扩展） | 路径长度不固定，最长可达 Root → TLD → SLD → 主机（4跳 + 权威查询） |
| **缓存优化**（TTL + 分层缓存） | 缓存期间数据可能陈旧（最终一致性，非强一致性） |
| **容错性**（每个 zone 至少2台 nameserver） | 缓存污染攻击（cache poisoning）风险 |
| **递归/迭代查询分离**（灵活部署） | 递归服务器成为 DoS 攻击放大器（DNS 放大攻击） |
| **轻量级协议**（UDP 主要，TCP 辅助） | 64KB UDP 响应限制（导致 EDNS0 扩展），大响应需要 TCP |

## Step 3 | 正确性与复杂性

### 正确性论证

RFC 1034 是一份**协议规范**而非理论论文，因此其"正确性"论证主要通过**逻辑一致性和工程经验**完成。

**论文/规范使用的论证方式**：

1. **不变量分析（Invariant Analysis）**：
   - 权威数据的来源：每个 RR 都有明确的"来源 zone"——zone 边界由 NS 记录明确定义
   - 委托链的完整性：从根到任意域名存在一条**唯一的委托路径**；如果路径断裂，查询就会失败（这是正确的失败信号）
   - 缓存新鲜度：由 TTL 控制；resolver 必须遵守 TTL（RFC 规范要求）

2. **查询语义的形式化描述**：
   - RFC 1034 用伪代码描述了 resolver 的查询算法，包括：
     - 查询顺序（先查缓存，再查权威）
     - 超时和重试策略
     - CNAME 跟随规则
   - 这些描述确保不同实现者的行为一致

3. **反熵与一致性**：
   - Zone transfer（AXFR/IXFR）确保 secondary nameserver 与 primary nameserver 保持同步
   - SOA 记录的序列号（serial）用于检测 zone 变更

**潜在正确性风险**（RFC 1034 已知晓但未完全解决）：

| 风险 | 描述 |
|------|------|
| **Cache Poisoning** | 攻击者向 resolver 注入伪造的权威记录；1980 年代未预见到这种攻击模式 |
| **Birthday Attacks on DNSSEC** | DNSSEC 引入后才暴露的安全问题 |
| **NXDOMAIN 放大** | Negative caching 可能被利用进行 DoS 攻击 |

### 复杂性分析

| 维度 | 分析 |
|------|------|
| **查询延迟（无缓存）** | 最坏情况：Root → TLD → SLD → Authoritative = **4 次往返**（RTT），加上最终查询 |
| **查询延迟（有缓存）** | TTL 为 0 时每次都需查询权威；TTL 较长时可直接从缓存返回（**O(1)**） |
| **缓存空间复杂度** | 取决于 resolver 内存大小；现代 resolver（BIND、Unbound）支持数万到数百万条缓存记录 |
| **Zone 文件大小** | 根 zone 约 2MB（包含所有 TLD NS 记录）；大型 zone（如 .com）可达数百 GB（次级 zone 数据） |
| **同步延迟（Zone Transfer）** | AXFR 完整传输一个大型 zone 可能需要数小时；IXFR 增量传输更高效 |
| **根服务器查询负载** | 实际仅约 13 组根服务器（A-M）；设计通过广泛缓存将根查询率降至每分钟数千次 |

**关键性能数字**（RFC 1034 原文）：
- 规范假设单次查询往返时间（RTT）在局域网 < 10ms，广域网 < 500ms
- 建议 resolver 实现**并发查询**（向多个可能知道答案的服务器同时发请求）

### 关键假设

1. **TTL 有效性**：所有参与者都遵守 TTL 规范（无恶意缓存不更新）
2. **时钟同步**：SOA 的刷新间隔依赖各 nameserver 时钟基本同步
3. **根服务器稳定性**：根服务器集合变化极小（1987 年以来只有极少的根服务器变更）
4. **安全假设（原始 DNS）**：网络参与者是可信的；未设计加密认证（DNSSEC 是后来在 2005 年 RFC 4033 补充的）
5. **域名唯一性**：委托授权确保每个节点只有唯一的权威来源

### 边界情况

| 边界情况 | 处理方式 |
|----------|----------|
| **域名不存在（NXDOMAIN）** | 返回"Name Error"；该响应可被 negative caching（RFC 2308） |
| **CNAME 链** | Resolver 必须跟随 CNAME直到找到最终 A 记录；RFC 规范限制 CNAME 链最多 1 次（即 CNAME 不能指向另一个 CNAME，现代放宽此限制但仍不鼓励） |
| **循环委托** | Zone A 委托 Zone B，Zone B 又委托回 Zone A → 查询进入死循环，resolver 必须通过 TTL 超时退出 |
| **Glue Records** | NS 记录指向子 zone 的主机（如 `ns.example.com` 在 `example.com` zone 中）→ 根/TLD 必须在答案中附带 glue record（A 记录）以避免"先有鸡还是先有蛋"问题 |
| **Wildcard Records** | `*.example.com` 匹配所有未明确列出的子域名；但 wildcard 展开规则复杂（RFC 4592 专门描述其语义陷阱） |

## Step 4 | 实验评估

### 实验设置

**RFC 1034/1035 是协议规范文档，本身没有传统意义上的"实验评估"**。

**验证方式**：

1. **BIND（Berkeley Internet Name Domain）实现**：
   - 1984 年由 UC Berkeley 开始开发；到 1987 年 RFC 1034/1035 发布时，BIND 已经成为 DNS 的主要实现
   - BIND 的广泛部署是 DNS 规范的事实验证——规范的可实现性由 BIND 证明，规范的清晰性由不同厂商的兼容实现证明

2. **DARPA 互联网测试**：
   - 在 RFC 1034/1035 发布前后，DNS 在 ARPANET/互联网早期部署中进行了小规模测试
   - 测试主要验证：不同厂商 nameserver 之间的互操作性、zone transfer 正确性、缓存有效性

**更完整的评估来自 DNS 部署后的运营数据**：

| 年份 | 评估内容 | 结论 |
|------|----------|------|
| **1990** | 根服务器负载测量 | 根查询率远低于预期（因为 TLD 服务器和 lower-level 缓存降低了根查询频率） |
| **1999** | DNS 缓存中毒攻击研究 | 发现 resolver 接受来自非权威源的附加记录（additional section）的安全性问题 |
| **2008** | Kaminsky DNS 缓存投毒漏洞 | 发现 UDP 源端口随机化可被绕过；推动 DNS 安全扩展（DNSSEC）的大规模部署 |
| **2010-至今** | DNSSEC 部署测量 | DNSSEC 验证率逐步提升（截至 2026 年约 25% 的 DNS 查询支持 DNSSEC） |

### 关键结果

**DNS 的成功验证：37 年零重大架构变更**：

截至 2026 年，DNS 的核心架构与 1987 年 RFC 1034/1035 的设计**几乎没有本质变化**：
- 层次化树形命名空间 → 仍在使用
- 根服务器系统（13 组 A-M）→ 仍在使用
- A/NS/MX/CNAME 记录类型 → 仍在使用
- UDP/TCP 端口 53 → 仍在使用
- Zone transfer 机制 → 仍在使用（仅增加 IXFR 增量协议）

唯一重大的架构级补充是 **DNSSEC（RFC 4033/4034/4035，2005 年）**——为 DNS 响应添加加密签名，解决缓存投毒和数据伪造问题。

### 局限性

1. **安全原语缺失**：1987 年的 DNS 设计**没有任何安全机制**——任何能拦截 DNS 查询的网络攻击者都可以伪造响应、投毒缓存。这直接导致了 2008 年 Kaminsky 漏洞，迫使整个互联网紧急部署 DNSSEC。
2. **隐私缺失**：DNS 查询是明文的（UDP 明文），ISP 和中间节点可以看到用户访问的所有域名。DoH（DNS over HTTPS，RFC 8484，2019）和 DoT（DNS over TLS，RFC 7858，2016）是应对措施。
3. **集中化趋势**：虽然 DNS 设计为分布式，但实际运营中出现了**公共递归 resolver 集中化**（Cloudflare 1.1.1.1、Google 8.8.8.8 等处理大量互联网查询），带来隐私和单点故障风险。
4. **Wildcard 语义混乱**：RFC 1034 对 wildcard 的描述不够精确，导致不同实现的语义不一致，直到 RFC 4592（2006）才完全澄清。

## Step 5 | 演化谱系（Evolution Lineage）

> **本节为任务明确要求，聚焦 DNS 的前驱、后续系统与工程落地**

### 前驱工作

| 前驱 | 年份 | 与 DNS 的关系 |
|------|------|---------------|
| **HOSTS.TXT** | 1970s | DNS 解决的直接问题——集中 hosts 文件管理的不可扩展性 |
| **RFC 952 / RFC 953** | 1980-1985 | 定义了 `hostname` 的语法规则（点分十进制格式、不超过 8 字符等），DNS 的命名规范直接继承自此 |
| **RFC 882 / RFC 883** | 1983 | DNS 的**首次草案**（后被废弃）；定义了与后来 RFC 1034 不同的数据结构（使用"domain name"概念的雏形但数据编码不同） |
| **ARPANET 命名实践** | 1970s | 早期互联网的命名约定，启发了层次化命名的思想 |
| **X.500 目录服务** | 1988（并行） | OSI 的全局目录标准；DNS 从中借鉴了目录分级的概念但采用了更轻量级的实现 |

### 直接后续系统（Direct Descendants）

| 系统 | 年份 | 建立在 DNS 之上的创新 |
|------|------|---------------------|
| **BIND（Berkeley Internet Name Domain）** | 1984-至今 | DNS 的第一个实现，至今仍是互联网最广泛使用的 nameserver 软件（估计 70%+ 的权威 DNS 服务器使用 BIND 或其衍生） |
| **DNSSEC（RFC 4033/4034/4035）** | 2005 | 为 DNS 响应添加公钥数字签名；解决缓存投毒和数据伪造问题 |
| **DoT / DoH** | 2016/2019 | DNS-over-TLS 和 DNS-over-HTTPS；将 DNS 查询加密以对抗中间人攻击和隐私泄露 |
| **mDNS / Bonjour** | 2000s | 基于 DNS 语义的多播本地服务发现协议；用于局域网内的零配置网络 |
| **DNS64 / NAT64** | 2007+ | IPv6 过渡技术；DNS64 合成 AAAA 记录以服务于纯 IPv4 内容 |
| **Authority Zones（RFC 2136 / ZONEMD）** | 1997/2020 | 为 DNS zone 数据添加消息摘要，防止未授权的 zone transfer |

### DNS 启发的 P2P 查找系统

| 系统 | 年份 | DNS 思想的传承 |
|------|------|--------------|
| **Chord（DHT）** | 2001（见 cs-learning #28） | 将 DNS 的层次化查找思想移植到 P2P 网络；一致性哈希环替代 DNS 树 |
| **Amazon Dynamo** | 2007（见 cs-learning #11） | 使用类似 DNS 委托的分区机制；向量时钟处理版本冲突 |
| **Google Chubby** | 2006（见 cs-learning #C1） | 使用 DNS 风格的层次化锁路径命名；被 GFS/Bigtable 用于选主 |

### DNS 的现实落地（Industrial Adoption）

- **所有互联网应用**都依赖 DNS 进行名称解析：Web 浏览器、电子邮件、移动应用
- **权威 DNS 服务**：Verisign（运营 .com/.net 根）、Cloudflare、Amazon Route 53、Google Cloud DNS
- **递归 DNS 服务**：ISP DNS（通常为本地缓存）、Cloudflare 1.1.1.1、Google 8.8.8.8、Quad9（9.9.9.9）
- **CDN 的核心**：Akamai、CloudFront 等 CDN 依赖 DNS 地理位置感知的负载均衡——DNS 返回距离用户最近的边缘节点 IP

## Step 6 | 个人理解

### 最重要的洞察

**DNS 最深刻的设计洞察不是任何具体的算法或数据结构，而是"委托授权 + 层次化树结构"所实现的无中央数据库全局协调**。

这个思想的核心是：**在一个需要全局一致性的系统中，如何在不依赖中央协调者的情况下，让每个参与者对"自己管理的那部分"拥有唯一发言权**。

DNS 的委托机制允许：
- **完全自主管理**：example.com 的所有者不需要向任何中央机构"请示"就能添加 `www.example.com`
- **信任传递**：根服务器不需要知道 `www.example.com` 的 IP，但它知道谁"应该知道"（.com 服务器）
- **自然聚合**：相近的域名自然聚合在同一权威下，便于管理

这种**信任层次结构**的设计哲学，深刻影响了后来的许多分布式系统——从 P2P DHT（Chord）到 Certificate Authority（CA）体系，再到 Kubernetes 的服务发现。

### 类比理解

**DNS = 的国际电话簿 + 层级分包快递系统**

- **HOSTS.TXT** = 有一本超级厚的全球电话簿，每周从纽约总部打印一次，然后通过船运分发到全球——当夏威夷收到新版本时，旧金山的主机已经更新了 3 次了
- **DNS** = 没有人拥有完整的电话簿，但每当你问"谁知道 Mrs. Smith 的电话？"，系统会告诉你"先问她住的城市，然后她住的街道，然后她家"——而且邻居会记住 Mrs. Smith 的号码，下次你问时直接告诉你，不用再走完整流程

### 疑问记录

- [ ] **DNS 的根服务器"信任固定 13 个节点"**——这是 DNS 架构中最接近"中心化"的部分。历史上根服务器变更（如 1998 年根签名密钥变更）极其罕见，但一旦出问题（如 2021 年 Fastly CDN 中断事件）影响巨大。为什么 1987 年选择了 13 个固定根服务器，而不是动态发现机制？
- [ ] **Negative caching 在 RFC 1034 原版中未定义**——这是一个重大遗漏，直到 1998 年 RFC 2308 才补充。原始设计中，不存在的域名每次都会被查询穿透到权威服务器，造成了不必要的负载。为什么这个明显必要的功能在最初设计时被遗漏了？
- [ ] **TTL 的动态调整**——现代 CDN 会将 TTL 设置为 5-60 秒（低缓存寿命）以实现快速故障转移，而传统静态网站可能设置为 86400 秒（1天）。TTL 的最优值取决于业务需求，但 resolver 无法区分"重要变化"和"无关变化"。是否有更好的机制让 zone 所有者传达"数据变化紧急程度"？
- [ ] **DNS 的隐私问题**：DoH/DoT 解决了传输层加密，但 DNS 本身的数据语义（你访问了哪些域名）本身就是敏感信息——即使内容加密，查询的元数据（DNS qname）仍然暴露。是否有完全隐私保护的名称解析方案？

## 原文摘录（精华句）

> "The domain system is intended to be used in the TCP/IP Internet. The protocols in the TCP/IP suite use the system to find addresses of hosts or gateways."
>
> — Mockapetllis, RFC 1034, 1987

> "The domain system has two principal components. The first is the name space – the set of all names that exist in the system. The second is the database – the resource records that contain information associated with the names."
>
> — Mockapetllis, RFC 1034, 1987

> "A domain is a subtree of the domain space. The root domain is the top of the tree, representing an empty name. The root domain is used to specify the top of the tree, but a name ending in a dot refers to an absolute name which ends at the root."
>
> — Mockapetllis, RFC 1034, 1987

> "A typical query might ask for the address associated with a host name. The query travels up the tree until it reaches a server that has the requested information, then travels back down."
>
> — Mockapetllis, RFC 1034, 1987
