# 29 | The Chubby Lock Service for Loosely-Coupled Distributed Systems（2006）分析报告

**作者**：Mike Burrows（Google Inc.）
**年份**：2006
**发表**：OSDI 2006
**分析日期**：2026-03-31

## TL;DR

- Chubby 用粗粒度锁服务替代"在每个系统内手写选主逻辑"，将 Paxos 共识封装在服务内部，对外暴露类文件系统接口；可靠性和可用性是首要目标，吞吐量是次要目标。
- 实际部署中最大用途是**命名服务**而非锁服务，这一意外比锁本身更能说明"小文件 + 强一致 + 事件通知"的组合威力。
- 论文价值在于完整记录了设计决策与实际使用之间的偏差，以及由此引发的多轮修改，是分布式系统工程演化的第一手文献。

## Step 1 | 背景与问题

### 1.1 核心需求

Google 内部大量分布式系统需要解决：

- 主副本选举（leader election）
- 服务发现与配置广播
- 粗粒度分布式互斥

在 Chubby 之前，各团队采用 ad hoc 方案或依赖人工干预，缺乏统一可靠的协调层。

### 1.2 为什么是锁服务而不是 Paxos 库

论文列出了四条理由：

| 理由 | 说明 |
|------|------|
| 开发习惯 | 开发者不会提前规划高可用；锁服务让现有程序只需增加 **2 条语句 + 1 个 RPC 参数**即可获得选主能力 |
| 服务发现 | Chubby 同时充当小文件服务，elected primary 可直接将自身地址写入 Chubby；减少客户端依赖的服务数 |
| 接口熟悉度 | 锁接口比 Paxos API 更易于程序员理解（尽管分布式锁仍有其微妙之处） |
| 法定人数外的单客户端前进 | 纯 Paxos 库要求 quorum；锁服务允许**单个客户端**持有锁并独立推进，降低客户端侧对副本数的要求 |

### 1.3 粗粒度 vs. 细粒度锁

| 维度 | 粗粒度（Chubby 的选择） | 细粒度 |
|------|------------------------|--------|
| 持有时长 | 小时/天（如选主后持续持有） | 毫秒/秒 |
| 锁获取频率 | 低 | 高 |
| 服务短暂不可用 | 客户端只需等待 | 大量事务失败 |
| 失败后锁存活 | 必须存活 | 可以丢弃 |

Chubby **只提供粗粒度锁**，这是最早的关键设计决策之一。

## Step 2 | 技术方案

### 2.1 系统结构

- Chubby cell = 通常 **5 个**副本（replica），分置不同机架以降低相关故障
- 副本通过分布式共识协议选举 master；master 需获得**多数派**投票，并获得"数秒内不另选他人"的 master lease
- 只有 master 处理读写；其他副本通过共识协议接收更新
- **写请求**：传播到所有副本，达到**多数派**确认后应答
- **读请求**：master 独立处理（在 master lease 未过期时安全）

### 2.2 命名空间与节点

- 典型路径形如 `/ls/foo/wombat/pouch`（ls = lock service；foo = cell 名，通过 DNS 解析）
- 只有文件和目录，统称"node"；无符号链接或硬链接
- 节点分两种：

| 类型 | 描述 |
|------|------|
| permanent | 显式删除前持久存在 |
| ephemeral | 无客户端打开时自动删除；用作存活指示器 |

- 每个节点可充当 advisory reader/writer lock
- 每节点维护 4 个单调递增的 64-bit 编号：instance number、content generation number、lock generation number、ACL generation number；另加 64-bit 内容校验和

### 2.3 锁与 Sequencer

- Advisory 锁（非强制）；支持 exclusive（写）和 shared（读）模式
- **Sequencer**：不透明字节串，包含锁名 + 模式 + lock generation number；持锁方传给下游服务器以验证锁的有效性，解决消息延迟导致的过期锁问题
- **Lock-delay**：持锁方故障（非正常释放）后，服务器延迟最多 **1 分钟**才允许其他客户端获取该锁；防止延迟消息或进程重启引发的错误

### 2.4 会话与 KeepAlive

- 客户端与 cell 之间维护 session；通过周期性 KeepAlive RPC 维持
- master 默认 lease 延长值为 **12 秒**（过载时可更大）
- 客户端收到 KeepAlive 回复后立刻发起下一次；"几乎总有一个 KeepAlive 在 master 侧阻塞"
- 事件通知和缓存失效通过 KeepAlive 捎带传输
- 若客户端本地 lease 超时：缓存清空并禁用，session 进入 **jeopardy** 状态
- **Grace period**（默认 45 秒）：在此期间若 KeepAlive 成功，session 得以恢复；否则视为过期

### 2.5 缓存

- 客户端在内存中维护一致的 write-through 缓存（包括文件不存在的 negative cache）
- 修改前：master 向所有可能缓存该文件的客户端发送**失效通知**，再执行修改
- 协议只下发失效，不下发更新（更新协议在极端情况下效率任意低）

### 2.6 故障切换（Fail-over）

新 master 恢复步骤（共 9 步，关键步骤）：

1. 选取新 epoch number（拒绝旧 epoch 的 handle）
2. 在内存中重建会话和锁结构
3. 保守地延长所有 lease
4. 向所有 session 发送 fail-over 事件
5. 等待客户端确认
6. 允许正常操作
7. 重建客户端请求的旧 handle
8. 约 1 分钟后删除无打开 handle 的 ephemeral 节点

Grace period 机制使 session 可以跨越超过正常 lease 时长的 master 切换而不丢失。

### 2.7 数据库实现演化

| 版本 | 实现 | 切换原因 |
|------|------|----------|
| 第一版 | 复制的 Berkeley DB（B-tree） | - |
| 当前版 | 自研简单数据库（write-ahead log + snapshot，Birrell 等人设计） | Berkeley DB 复制代码较新、用户少，风险敞口过大 |

日志通过共识协议分发；快照写入 GFS（**不同楼**，避免循环依赖）。

## Step 3 | 正确性保障

### 3.1 Sequencer 解决过期锁问题

```
持锁方 C 获取锁 L，得到 sequencer S = {name, mode, gen=7}
C 向服务器发送请求，附带 S
服务器调用 CheckSequencer(S)
  若 gen != 当前 lock gen -> 拒绝（C 已失去锁）
  若 gen == 当前 lock gen -> 接受
```

这将"锁是否仍有效"的判断权交给**接收方服务器**，而非依赖消息到达顺序。

### 3.2 Master Lease 保证读正确性

master 持有 lease 期间，不会有其他节点被选为 master。因此 master 独立处理读请求是安全的，不需要 quorum read。

### 3.3 Lock-delay 防护区间

```
T0: 客户端 A 持锁，网络延迟导致消息 M 尚未到达服务器
T1: A 会话超时，锁被释放
T2: 服务器等待 lock-delay（最多 1 分钟）
T3: 延迟 M 到达服务器 -> 若 T3 < T2，锁仍由 A 持有，M 被接受
     若 T3 > T2，新持锁方已存在，M 通过 sequencer 被拒绝
```

### 3.4 形式化保障小结

| 性质 | 保障机制 |
|------|----------|
| 安全性（同时只有一个 master） | Paxos 多数派 + master lease |
| 活性（最终选出 master） | Paxos 共识协议 |
| 读一致性 | Master lease 未过期时本地读安全 |
| 锁有效性验证 | Sequencer + lock generation number |
| 会话存活跨 fail-over | Grace period (45s) |

## Step 4 | 实验与生产数据

### 4.1 生产环境快照（论文 Section 4.1）

| 指标 | 值 |
|------|----|
| 距上次 fail-over 时间 | 18 天 |
| 上次 fail-over 持续时间 | 14 秒 |
| 直连活跃客户端 | 22,000 |
| 通过 proxy 的额外客户端 | 32,000 |
| 打开的文件数 | 12,000 |
| 其中命名相关文件 | 60% |
| 客户端缓存条目 | 230,000 |
| 被缓存的不同文件数 | 24,000（平均每文件被 ~10 个客户端缓存） |
| 独占锁 | 1,000 |
| 共享锁 | 0 |
| 存储文件数 | 22,000 |
| 0-1KB 文件占比 | 90% |
| 1KB-10KB 文件占比 | 10% |
| >10KB 文件占比 | 0.2% |

### 4.2 RPC 流量分布

| RPC 类型 | 占比 |
|----------|------|
| KeepAlive | 93% |
| GetStat | 2% |
| Open | 1% |
| CreateSession | 1% |
| GetContentsAndStat | 0.4% |
| SetContents（写） | 680 ppm |
| Acquire（获取锁） | 31 ppm |

写操作仅占 680 ppm，锁获取仅 31 ppm；与粗粒度锁的设计预期一致。

### 4.3 延迟

- 平均请求延迟：在接近过载前，始终保持在**毫秒的一小部分**以内
- 过载触发点：活跃 session 超过约 90,000 时
- 客户端侧 RPC 读延迟：同数据中心 < 1ms；两极之间 ~250ms
- 写延迟：数据库日志额外增加 5-10ms；若近期故障客户端缓存了该文件，可达数十秒

### 4.4 可用性

- 约 700 cell-days 内发生 61 次中断；其中 52 次 < 30 秒
- 大多数应用对 < 30 秒的中断无显著影响
- 数据丢失：数十 cell-years 内仅 6 次，其中 4 次为数据库软件错误，2 次为操作员错误；无硬件导致的数据丢失

### 4.5 扩展方式

| 机制 | 效果 |
|------|------|
| 每数据中心独立 cell | 隔离故障域 |
| 增大 lease 时长（12s -> ~60s） | 降低 KeepAlive 频率 |
| 客户端缓存 | 减少读 RPC |
| Proxy 服务器（写作时未投产） | 可将 KeepAlive 流量降低 Nproxy 倍（可能达 10,000 倍以上） |
| 名字空间分区（写作时未投产） | 按目录 hash(D) mod N 分片 |

## Step 5 | 影响力分析

### 5.1 Google 内部直接影响

- **GFS**：用 Chubby 锁选举 GFS master
- **Bigtable**：用 Chubby 选举 master、发现 tablet server、允许客户端定位 master；以 Chubby 为分布式数据结构的根
- **MapReduce**：作为公共集合点（rendezvous）
- **最终地位**（论文 Section 6）："Chubby 已成为 Google 的主要内部命名服务"——这是设计时的意外

### 5.2 工业界影响

- 直接启发 Apache ZooKeeper（开源替代；2010 年 ATC 论文）
- ZooKeeper -> 启发 etcd -> Kubernetes 控制平面
- "锁服务 + 小文件 + 事件通知"的组合成为分布式协调层的标准模式

### 5.3 与 ZooKeeper 的关键差异

| 维度 | Chubby | ZooKeeper |
|------|--------|-----------|
| 接口抽象 | 类文件系统 + 显式锁 API | 树形 znode + watch，无显式锁 |
| 一致性协议 | Paxos | ZAB（与 Paxos 等价） |
| 读语义 | master 本地读（lease 保护） | 可从任意副本读（较弱语义），或 sync() 强制 |
| 副本数 | 典型 5 个 | 典型 3 或 5 个 |
| Grace period | 有（45s） | 无直接等价机制 |

## Step 6 | 个人理解

### 6.1 最重要的工程洞察

**实际用途与设计意图的偏差是这篇论文最有价值的部分。**

Chubby 被设计为锁服务，但最终成为 Google 的主要命名服务。根本原因是：

- DNS 依赖 TTL（被动失效）；Chubby 使用主动失效通知
- 同等硬件上，Chubby master 可服务 90,000 个客户端，而 DNS 服务器仅约 50,000 req/s
- "3,000 个进程 + 60s TTL + 二次方 DNS 查找 = 每秒 150,000 次 DNS 查询"的问题，Chubby 通过批量名字条目（100 个相关进程一条记录）解决

### 6.2 关键教训

| 教训 | 来源 | 影响 |
|------|------|------|
| 开发者不考虑高可用 | Section 4.6 | Chubby 的存在意义本身：降低"拥有高可用系统"的门槛 |
| 细粒度锁实际从未被需要 | Section 4.6 | 最初设计讨论中被考虑的功能，生产环境中完全不需要 |
| TCP KeepAlive 在网络拥塞时引发大量 session 丢失 | Section 4.6 | 被迫切换为 UDP 发送 KeepAlive |
| 新 session 必须写入 DB 导致大量进程同时启动时过载 | Section 4.4 | 重新设计：仅在首次修改/获取锁/打开 ephemeral 文件时才写入 DB |
| 一个客户端 1.5MB 文件每次用户操作都全量重写 | Section 4.5 | 引入 256KB 文件大小限制 |

### 6.3 粗粒度锁的适用性

论文中 31 ppm 的 Acquire 比例印证了一个反直觉结论：**分布式锁在实践中被极少使用，但它的存在极为关键**。选主锁可能一天只获取一次，但这一次决定了整个系统的主副本状态。

## 关键权衡表

| 维度 | Chubby 的选择 | 代价 |
|------|---------------|------|
| 可用性 vs. 性能 | 可用性优先；吞吐量是次要目标 | RPC 延迟受限于 Paxos 轮次 + 数据库日志（5-10ms） |
| 粗粒度锁 vs. 细粒度锁 | 仅支持粗粒度 | 不适合高频细粒度同步场景 |
| 单 master 读 vs. 任意副本读 | Master 读（lease 保护） | Master 是瓶颈；单 cell 最多约 90,000 客户端 |
| 缓存失效 vs. 缓存更新 | 只下发失效 | 客户端缓存 miss 时须重新读取，有一次 RPC 开销 |
| 服务合并 vs. 服务分离 | 锁+文件+命名+事件合一 | 单个系统承担多种语义；与 Boxwood 三服务分离形成对比 |
| 主动失效通知 vs. TTL | 主动失效（对比 DNS TTL） | 需要 master 跟踪所有缓存客户端；230,000 缓存条目 |

## 局限与风险

- 单 cell master 是单点（虽有 lease + fail-over 保护），fail-over 持续时间从数秒到 30 秒以上不等
- Fail-over 代码"是有趣 bug 的富矿"（论文原文），测试覆盖率天然低于正常路径
- 不适合细粒度/高频锁场景；文件大小硬上限 256KB
- 跨数据中心延迟（~250ms）使远程 cell 读写代价高昂；GFS 和 Bigtable 的 Chubby 调用必须走本地 cell
- 客户端滥用（如 publish/subscribe 误用、缺乏缓存的轮询循环）会快速消耗 cell 容量；需要使用审查流程控制
