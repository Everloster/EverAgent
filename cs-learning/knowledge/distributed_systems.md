# 分布式系统基石：MapReduce编程模型

## 核心论文背景
**作者**：Jeffrey Dean & Sanjay Ghemawat | **机构**：Google | **年份**：2004
**引用量**：20000+（分布式系统被引最多论文之一）

---

## 问题与解决方案

### 2004年Google的困境
- 数据规模：数十亿网页文档，数TB级原始数据
- 需求：定期重建全网索引，计算PageRank等复杂算法
- 硬件：数千台廉价商用服务器，非超算
- 挑战：数据分片、故障恢复、机器间协调、负载均衡全部需要手工解决

每个新任务都要重复实现分布式框架，投入巨大。

### 核心洞察
**分离关注点**：数据处理逻辑与分布式执行机制完全可分离。
- 用户描述"做什么"（Map、Reduce函数）
- 框架负责"如何在分布式环境中做"（调度、容错、通信）

## MapReduce编程模型

### 基本抽象
```
输入：key-value对流
     ↓
Map函数（用户定义）：(k1,v1) → list(k2,v2)
     ↓ [自动分组排序]
Reduce函数（用户定义）：(k2,list(v2)) → list(v3)
     ↓
输出：key-value对流
```

**WordCount示例**：
- Map：文档→(单词,1)流
- Shuffle：框架按单词分组排序
- Reduce：(单词,[1,1,1,…]) → (单词,计数)

### 执行流程（7步）
1. **Split输入**：M个16-64MB分片，启动程序副本
2. **Master分配**：M个Map、R个Reduce任务分配给Workers
3. **Map执行**：读输入分片，执行Map函数，缓存中间结果
4. **本地持久化**：中间结果写入本地磁盘，分为R个区
5. **Reduce读取**：通过RPC从Map Worker读取，按key排序
6. **Reduce执行**：对每个唯一key调用Reduce函数
7. **完成**：所有任务完成，返回结果

## 容错机制（核心工程价值）

### Worker故障
- **检测**：Master定期ping，无响应标记failed
- **Map任务**：必须重做（中间数据在本地磁盘，已失效）
- **Reduce任务**：无需重做（输出写入GFS全局文件系统）

关键设计：Reduce结果持久化，Map结果非持久化——直接影响故障恢复策略。

### 慢节点处理（Straggler Mitigation）
**问题**：某台机器因磁盘/CPU争抢而极慢，整个job需等待。

**解方**：备份任务（Backup Tasks）
- 接近完成时，Master在其他空闲机器备份执行进行中的任务
- 原始或备份任一先完成即标记完成
- 实测：5%资源开销换来44%延迟减少

## 优化技术

### Combiner（局部预聚合）
Map端本地先进行Reduce操作，大幅减少网络传输数据量。
- 示例：WordCount中，无Combiner则网络传输百万条(word,1)；有Combiner则只传(word,count)

### 自定义分区函数
默认hash(key) mod R。用户可覆盖，如按URL域名分区。

### 顺序保证
给定分区内的键值对按键递增顺序处理。

### 跳过坏记录
若特定输入导致Worker崩溃，反复检测该记录序号后续跳过。

## 性能数据

### Grep任务（扫描10^10个100字节记录）
| 阶段 | 数据量 | 时间 |
|------|--------|------|
| 1800台机器，Map峰值速率 | 30 GB/s | 80秒 |
| **总时间** | **1TB**  | **~150秒** |

### TeraSort（10^10个100字节记录排序）
| 阶段 | 时间 |
|------|------|
| Map | 200秒 |
| Shuffle/Sort | 数分钟 |
| Reduce | 200秒 |
| **总计** | **~891秒（15分钟）** |

**吞吐量**：6.26 GB/s聚合写入速率。

## 历史影响

### 开源生态链
```
MapReduce (2004, Google 内部)
       ↓
Hadoop MapReduce (2006, Doug Cutting)
       ├→ HDFS（GFS开源实现）
       ├→ Hive (2008, Facebook)——SQL接口
       └→ Pig (2008, Yahoo)——数据流语言
```

### 被超越的轨迹
- **中间结果反复落盘低效** → Spark (2012)：内存计算+RDD
- **不支持流式** → Flink (2014)：统一批流
- **低延迟交互查询** → BigQuery (2010)：列式存储
- **图计算效率** → Pregel (2010)：BSP模型

2014年，Google内部基本停用Hadoop MapReduce，转向Dataflow。2016年，Cloudera宣布进维护模式。但MapReduce的**思想**（分离逻辑与执行、容错设计）在Spark、Flink等中延续。

## 设计原则总结

| 原则 | 体现 |
|------|------|
| **幂等性** | Map/Reduce可安全重跑，输出确定性 |
| **无副作用** | Map/Reduce是纯函数，容错简化 |
| **宽依赖** | Shuffle是全对全通信，性能瓶颈 |
| **推测执行** | 备份任务应对Straggler，现已标配 |
| **数据本地性** | 计算靠近存储，减少网络开销 |

---

## 当代意义

- **大数据基础设施**：思想影响从Hadoop到现代Spark/Flink生态
- **云计算模型**：无状态计算+持久化存储的分离，Serverless的原型
- **分布式设计原则**：幂等性、容错、宽依赖等概念仍是设计核心
- **工程实践**：2004年的MapReduce框架设计至今仍有启发价值
