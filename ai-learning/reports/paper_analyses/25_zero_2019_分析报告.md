## 🎯 一句话总结

> ZeRO 通过将模型状态（优化器状态、梯度、参数）在数据并行进程间分片，消除显存冗余，使单GPU显存消耗与数据并行度成反比，同时保持与数据并行基本一致的通信效率。

---

## Step 1: 背景与问题（WHY）

### 1.1 问题：大模型训练的显存瓶颈

训练超大模型（如数十亿到数万亿参数）面临的核心挑战是**GPU 显存不足以容纳完整模型**。

- **现状**：即使是最先进的 GPU（如 32GB V100），当模型参数量超过 14 亿时，训练也会因显存不足而崩溃
- **问题本质**：现有的分布式训练方法在显存效率和计算效率之间存在不可调和的矛盾

### 1.2 Prior Work：既有方案的缺陷

| 方法 | 原理 | 缺陷 |
|------|------|------|
| **数据并行 (DP)** | 每 GPU 复制完整模型，独立计算梯度，通过 AllReduce 同步 | 每个 GPU 都需要完整模型副本，显存冗余严重 |
| **模型并行 (MP)** | 将模型横向/纵向切分到多设备 | 计算粒度太细，层间通信密集，跨节点效率极低（<5% 峰值） |
| **流水线并行 (PP)** | 将模型按层切分，形成流水线 | 显存Bubble问题，通信开销大 |

### 1.3 作者的洞察

作者发现训练过程中的显存消耗主要分为两类：

1. **模型状态（Model States）**：优化器状态、梯度、参数，占据最大显存
2. **剩余状态（Residual States）**：激活值、临时缓冲区、内存碎片

传统数据并行需要**静态维护所有模型状态**，但训练时并不总是需要全部状态——这意味着存在优化空间。

---

## Step 2: 技术方案（HOW）

### 2.1 核心思想：用通信换显存

ZeRO 的核心思想是：**不在每 GPU 上复制完整模型状态，而是将模型状态分片（Partition）到不同的 GPU 上**。

通过动态通信调度，在需要时按需获取完整状态，从而在保持数据并行计算粒度的同时，实现模型并行的显存效率。

### 2.2 ZeRO-DP 三阶段优化

ZeRO-DP 提供三个递进阶段的优化：

#### Stage 1：优化器状态分片（$P_{os}$）

- **做法**：只保留 1/N 的优化器状态（Adam 的 momentum 和 variance）
- **显存**：每 GPU 从 $12\Psi$ 降至 $4\Psi$（减少 4 倍）
- **通信**：与标准 DP 相同（需对优化器状态进行 all-gather）

#### Stage 2：优化器状态 + 梯度分片（$P_{os+g}$）

- **做法**：在 Stage 1 基础上，对梯度也进行分片
- **显存**：每 GPU 从 $12\Psi$ 降至 $2\Psi$（减少 8 倍）
- **通信**：与标准 DP 相同

#### Stage 3：全部分片（$P_{os+g+p}$）

- **做法**：在 Stage 2 基础上，对模型参数也进行分片
- **显存**：每 GPU 降至 $\Psi/N$（N 个 GPU 减少 N 倍）
- **通信**：增加约 50%

### 2.3 核心公式推导

#### 混合精度训练下的模型状态显存

假设模型参数量为 $\Psi$（以参数计），使用混合精度训练（fp16 用于前向/反向，fp32 用于优化器状态）：

| 组件 | 精度 | 显存占用 |
|------|------|----------|
| 模型参数 | fp16 | $2\Psi$ bytes |
| 梯度 | fp16 | $2\Psi$ bytes |
| Adam 动量 (m) | fp32 | $4\Psi$ bytes |
| Adam 方差 (v) | fp32 | $4\Psi$ bytes |
| fp32 主权重备份 | fp32 | $4\Psi$ bytes |
| **总计** | | **$16\Psi$ bytes** |

论文中采用简化计算（考虑梯度与参数可复用），实际为 $12\Psi$ bytes。

#### ZeRO 各阶段显存公式

设 $N_D$ 为数据并行度（GPU 数量），$\Psi$ 为参数量：

| 阶段 | 每 GPU 显存公式 | 相对减少 |
|------|-----------------|----------|
| 标准 DP | $12\Psi$ | 1x |
| ZeRO-1 ($P_{os}$) | $4\Psi + \frac{8\Psi}{N_D}$ | ~4x |
| ZeRO-2 ($P_{os+g}$) | $2\Psi + \frac{10\Psi}{N_D}$ | ~8x |
| ZeRO-3 ($P_{os+g+p}$) | $\frac{12\Psi}{N_D}$ | Nx |

**简化理解**：
- ZeRO-1：每 GPU 只保留 1/N 的优化器状态
- ZeRO-2：每 GPU 只保留 1/N 的优化器状态 + 梯度
- ZeRO-3：每 GPU 只保留 1/N 的参数

### 2.4 通信量分析

| 方法 | 通信量 |
|------|--------|
| 标准 DP | $2\Psi$ (参数/带宽) |
| ZeRO-1/2 | $2\Psi$ (与 DP 持平) |
| ZeRO-3 | $3\Psi$ (增加 50%) |

### 2.5 伪代码：ZeRO-3 训练流程

```python
# ZeRO-3 简化伪代码
class ZeRO_Optimizer:
    def __init__(self, model, num_gpus):
        self.param_partitions = partition_model_params(model, num_gpus)
        self.optimizer_states = partition_optimizer_states(model, num_gpus)
        
    def forward(self, input_data, partition_id):
        # 按需获取当前 partition 需要的参数
        all_params = all_gather(self.param_partitions[partition_id])
        local_output = forward_with(all_params, input_data)
        return local_output
    
    def backward(self, loss, partition_id):
        # 计算局部梯度
        local_grad = backward_gradient(loss)
        # 对梯度分片（Scatter）
        partitioned_grad = scatter(local_grad, num_gpus)
        # All-reduce 得到全局梯度均值
        global_grad = all_reduce(partitioned_grad)
        # 更新本地参数分片
        self.update_partition(global_grad, partition_id)
        
    def update_partition(self, grad, partition_id):
        # 只更新当前 GPU 负责的参数分区
        self.optimizer_states[partition_id].step(grad)
        # 将更新后的参数广播给其他 GPU
        broadcast(self.param_partitions[partition_id])
```

### 2.6 ZeRO-R：剩余状态优化

除了模型状态，ZeRO 还提出 ZeRO-R 优化剩余显存：

1. **激活分片（$P_a$）**：对激活检查点进行分片，减少激活显存
2. **固定缓冲区（CB）**：使用固定大小缓冲区，避免动态分配开销
3. **内存碎片管理（MD）**：主动整理碎片，避免显存峰值

## Step 3: 证据与论证

### 3.1 理论论证

论文提供了严格的数学证明：

1. **显存下界**：ZeRO-3 达到每 GPU 显存的下界 $\Psi/N$，即不可能更少
2. **通信上界**：ZeRO-1/2 的通信量等于标准 DP，ZeRO-3 仅增加 50%
3. **计算效率**：保持数据并行的计算粒度（粗粒度），避免模型并行的细粒度通信瓶颈

### 3.2 关键假设

- 通信带宽充足：假设节点内/节点间有足够带宽
- 动态调度：训练不同阶段按需获取参数，减少同时需要的数据量

## Step 6: 个人理解

### 6.1 核心洞见

ZeRO 的本质是**在时间（通信）和空间（显存）之间做权衡**：

- 传统 DP：用空间换时间（每 GPU 保存完整副本，无通信等待）
- 传统 MP：用时间换空间（切分参数，细粒度通信换取显存）
- ZeRO：动态调度，部分时间通信换取大部分空间节省

### 6.2 最难理解的部分

**动态通信调度**：ZeRO 需要精确调度何时获取/释放参数，这涉及：
- 前向传播时的参数预取
- 反向传播时的梯度同步
- 参数更新后的广播时序

### 6.3 实现难点

1. **通信与计算重叠**：需要精心设计流水线隐藏通信延迟
2. **分布式优化器**：修改优化器使其支持分片状态下的参数更新
3. **容错处理**：部分 GPU 失败时的状态恢复
