# #25 | ZeRO: Memory Optimizations Toward Training Trillion Parameter Models 分析报告

**作者**：Samyam Rajbhandari, Jeff Rasley, Olatunji Ruwase, Yuxiong He（Microsoft Research）
**年份**：2019（arXiv）/ 2020（SC20 最佳论文候选）
**发表**：arXiv:1910.02054, ACM/IEEE SC 2020
**分析日期**：2026-03-30
**阅读难度**：⭐⭐⭐⭐（需熟悉分布式训练、GPU内存架构）

---

## TL;DR（一段话总结）

ZeRO（Zero Redundancy Optimizer）直指大模型训练的核心瓶颈——GPU 显存。传统数据并行在每张 GPU 上完整复制模型参数、梯度和优化器状态（Adam 约需 16 倍参数量的显存），造成巨大冗余。ZeRO 通过三阶段渐进式分区——先分优化器状态（Stage 1，4x 节省）、再分梯度（Stage 2，8x 节省）、最后分参数（Stage 3，与 GPU 数成比例节省），在保持数据并行的通信效率的同时，实现了接近模型并行的内存效率。在 400 块 V100 上训练 100B 模型时，ZeRO 达到 15 Petaflops 吞吐量，相比基线提升 8 倍。

---

## Step 1 | 背景与问题

### 历史节点

2019 年，GPT-2（1.5B）、T5-11B、Turing-NLG（17B）相继发布。参数量每几个月翻一个量级，但 GPU 显存（V100 32GB）增长缓慢。训练 GPT-3（175B）需要海量 GPU 集群——如何让有限的 GPU 算力训练出更大的模型，成为最紧迫的工程问题。

### 核心问题

> **如何在不显著增加通信开销的前提下，消除数据并行中的显存冗余，使 N 张 GPU 能训练 N 倍大的模型？**

### 前人方案的不足

| 方案 | 问题 |
|------|------|
| 纯数据并行（Data Parallelism） | 每GPU复制全部状态，N卡只能训练1×模型 |
| 模型并行（Pipeline/Megatron-LM） | 通信开销大、GPU利用率低、实现复杂 |
| 梯度压缩（1-bit SGD） | 牺牲精度，收敛不稳定 |
| CPU Offload（Gpipe-Offload） | CPU-GPU 带宽瓶颈严重 |

### 论文的核心主张

> **一句话**：数据并行的显存冗余是可消除的——通过分区优化器状态、梯度和参数，ZeRO 实现了近乎零冗余，同时保持与数据并行相当的通信效率。

## Step 2 | 技术方案

### 核心思想

传统数据并行中，每张 GPU 都存有完整的：
- **模型参数** Φ（4 bytes/param, fp32）或（2 bytes/param, fp16）
- **梯度** ∇Φ（与参数同大小）
- **优化器状态**（Adam 需要 FP32 参数副本 + 动量 m + 方差 v，共 12 bytes/param）

一个 1B 参数的模型，仅 Adam 优化器就需要约 16GB 显存（1B × 16 bytes），V100 32GB 只能放一个 2B 模型。

ZeRO 的洞察：**这些状态在所有 GPU 上是完全相同的，何必全部复制？** 只要让每张 GPU 存一部分，需要时通过通信获取即可。

### 关键设计

**内存消耗公式**

以训练 N 参数模型（Adam、FP16 混合精度）为例，单 GPU 需要的显存：

```
传统数据并行:
  Ψ_dp = Ψ_os + Ψ_g + Ψ_p
       = 12Ψ + 2Ψ + 2Ψ  (Adam fp32参数+动量+方差, fp16梯度, fp16参数)
       = 16Ψ

ZeRO-1 (P_os):
  Ψ_{os} = 12Ψ/N + 2Ψ + 2Ψ = 4Ψ + 12Ψ/N  → ≈ 4Ψ (N足够大时)

ZeRO-2 (P_os + P_g):
  Ψ_{os+g} = 12Ψ/N + 2Ψ/N + 2Ψ = 2Ψ + 14Ψ/N  → ≈ 2Ψ

ZeRO-3 (P_os + P_g + P_p):
  Ψ_{os+g+p} = (12 + 2 + 2)Ψ/N = 16Ψ/N
```

**Stage 1：优化器状态分区（P_os）**

- 每个 GPU 只负责 1/N 的优化器状态（m_i, v_i）
- 梯度和参数仍全量复制
- 反向传播后：每个 GPU 计算 ∇Φ → Reduce-Scatter 聚合到对应分区 → 各 GPU 用自己的优化器状态更新对应分区的参数
- **内存节省**：12Ψ → 12Ψ/N，约 4 倍

**Stage 2：梯度分区（P_os + P_g）**

- 在 Stage 1 基础上，梯度也分区存储
- 反向传播后：Reduce-Scatter 直接将梯度放到对应 GPU，无需先 All-Reduce 再丢弃
- **内存节省**：额外 2Ψ → 2Ψ/N，约 8 倍总计

**Stage 3：参数分区（P_os + P_g + P_p）**

- 模型参数也分片，每个 GPU 只存 1/N 的参数
- 前向传播时：通过 All-Gather 获取完整参数 → 计算 → 丢弃
- 反向传播时：同理获取完整参数计算梯度
- **内存节省**：16Ψ → 16Ψ/N，与 GPU 数成线性比例

**通信量对比**

| 阶段 | 额外通信量 | 相比数据并行 |
|------|-----------|-------------|
| Stage 1 | 0 | 无额外开销 |
| Stage 2 | 0 | Reduce-Scatter 替代 All-Reduce，通信量相同 |
| Stage 3 | 1.5× | 需额外 All-Gather 参数，但避免模型并行的层间通信 |

### 权衡分析

| 获得 | 牺牲 |
|------|------|
| 内存与 GPU 数成反比 | Stage 3 需要额外 All-Gather 通信 |
| 保持数据并行的易用性 | Stage 3 实现复杂度较高 |
| 超线性加速（N↑通信↓） | 对网络带宽有更高要求 |
| 兼容现有训练代码 | 增加了通信同步点 |

## Step 3 | 理论证明（核心分析）

### 内存上界证明

**引理**：ZeRO-Stage k 的单 GPU 最大内存消耗为：

```
Stage 1: Ψ_max = 2Ψ + (4 + 12/N_d)Ψ
Stage 2: Ψ_max = (2 + 2/N_d)Ψ + (4 + 12/N_d)Ψ  
Stage 3: Ψ_max = (2 + 2/N_d + 2/N_d)Ψ + (4 + 12/N_d)Ψ
```

其中 N_d 为数据并行度（GPU 数），附加项为临时激活值和通信缓冲区。

### 通信复杂度

| 操作 | 数据并行 | ZeRO-1 | ZeRO-2 | ZeRO-3 |
|------|---------|--------|--------|--------|
| All-Reduce 梯度 | 2Ψ | 2Ψ | — | — |
| Reduce-Scatter 梯度 | — | — | Ψ | Ψ |
| All-Gather 参数 | — | — | — | (N_d-1)/N_d × 2Ψ × L |
| All-Gather 梯度 | — | — | — | (N_d-1)/N_d × 2Ψ × L |

其中 L 为层数。ZeRO-3 的通信量约 1.5 倍于数据并行，但远低于模型并行（Pipeline parallelism 需要每层同步）。

### 关键假设

- 高带宽互联（InfiniBand/NVLink）
- Adam 优化器（或类似的二阶优化器）
- 混合精度训练（FP16/BF16）
- 同构 GPU 集群

## Step 4 | 实验评估

### 实验设置

- **硬件**：256-400 块 NVIDIA V100（32GB）
- **网络**：InfiniBand HDR（200 Gbps）
- **模型**：GLM-130B 基线、自研 100B+ 模型
- **基线**：PyTorch DDP、Megatron-LM
- **评价指标**：TFLOPs/GPU、训练吞吐量、最大可训练模型规模

### 关键结果

| 配置 | 方法 | 模型规模 | TFLOPs/GPU | 总吞吐量 |
|------|------|---------|------------|---------|
| 64 V100 | PyTorch DDP | 1.5B | 18 | 1.15 Petaflops |
| 64 V100 | ZeRO-2 | 7.5B | 38 | 2.4 Petaflops |
| 64 V100 | ZeRO-3 | 7.5B | 30 | 1.9 Petaflops |
| 400 V100 | ZeRO-2 | 100B | 38 | 15 Petaflops |
| 400 V100 | ZeRO-3 | 100B | 33 | 13 Petaflops |

**关键观察**：
1. **超线性加速**：从 64 → 400 GPU，单 GPU TFLOPs 反而上升（因为模型更大，计算/通信比更高）
2. ZeRO-2 在通信效率上优于 ZeRO-3（无额外 All-Gather）
3. 400 GPU ZeRO-2 训练 100B 模型：15 Petaflops，是当时公开报道的最大规模
4. 模型并行（Megatron-LM）在相同规模下 TFLOPs 仅为 ZeRO 的一半

### 局限性

- 实验仅用 V100，未测试 A100/H100 上的表现
- 未与 FSDP（PyTorch 原生实现）对比
- 激活值重计算（activation checkpointing）的影响未独立分析
- 异构集群场景未涉及

## Step 5 | 影响力

### 直接影响

| 系统/工具 | 关系 |
|-----------|------|
| **Microsoft DeepSpeed** | ZeRO 是 DeepSpeed 框架的核心引擎 |
| **Hugging Face Accelerate** | 集成 ZeRO-2/3 作为默认大模型训练方案 |
| **PyTorch FSDP** | 受 ZeRO 启发，PyTorch 官方实现参数分片（Fully Sharded Data Parallel） |
| **Megatron-DeepSpeed** | NVIDIA 与微软合作，将 Megatron 张量并行与 ZeRO 结合 |
| **BLOOM（176B）** | 使用 ZeRO-3 + DeepSpeed 在 384 块 A100 上训练 |
| **LLaMA 训练** | Meta 的 LLaMA 系列使用了类似 ZeRO 的参数分片策略 |

### 学术影响

- 开创了"消除数据并行冗余"这一研究方向
- ZeRO-Offload（SC 2021）和 ZeRO-Infinity（ASPLOS 2022）将思路扩展到 CPU 和 NVMe
- 推动了"通信效率 vs 内存效率"这一 trade-off 的系统化研究
- 成为大规模模型训练的事实标准（de facto standard）

### 批评与争议

1. **通信瓶颈**：ZeRO-3 的 All-Gather 在低带宽网络下可能成为瓶颈
2. **与 FSDP 的重叠**：PyTorch FSDP 本质上是 ZeRO-3 的重新实现，引发"谁是原创"的讨论
3. **Pipeline 并行兼容性**：ZeRO 与 Pipeline 并行的结合（3D 并行）需要额外工程
4. **微调场景的浪费**：ZeRO 对全量训练优化显著，但 LoRA/QLoRA 等参数高效微调方法使其在微调场景下不那么必要

## Step 6 | 个人理解

### 最重要的洞察

ZeRO 最精妙的洞察在于：**数据并行中最大的浪费不是计算，而是存储冗余**。所有 GPU 做着完全相同的优化器更新，却各自存了一份完整的状态。这不是硬件限制导致的，纯粹是软件设计上的"偷懒"——图省事全部复制。ZeRO 用分区+通信的方式，几乎零代价地消除了这个冗余。

另一个值得注意的设计哲学：**渐进式优化**（Stage 1→2→3）。不需要一步到位，用户可以根据网络条件选择合适的阶段。Stage 1 零额外通信，Stage 2 通信量不变，Stage 3 才需要额外通信。这种"免费→等价→付费"的梯度设计，大大降低了部署门槛。

### 类比理解

想象 10 个人（GPU）合作翻译一本 10000 页的书（模型参数）。

**传统数据并行**：每个人各自复印了一本完整的书（10 本！），各自翻译一页后汇总。书占桌子太大（显存不够）。

**ZeRO-1**：每个人只保留自己负责那 1000 页的翻译笔记（优化器状态），但书还是要完整放着。

**ZeRO-2**：连翻译草稿也只保留自己那份，共享草稿用完就传走。

**ZeRO-3**：连原书也只留 1/10，需要哪页就问邻居借来翻一下，看完还回去。

### 疑问记录

- [ ] ZeRO-3 的 All-Gather 与 Pipeline Parallel 的前向激活传递如何叠加？是否会形成通信热点？
- [ ] ZeRO-Offload 将优化器状态放到 CPU 时，PCIe 带宽（~16 GB/s vs NVLink ~300 GB/s）对训练速度的影响有多大？
- [ ] 在非 Adam 优化器（如 SGD、AdaFactor）下，ZeRO 的收益有何变化？
