## 🔍 层次一：5岁小孩也能懂的类比

**类比故事**：想象一所医院里有 8 位专科医生——神经科、骨科、心脏科……每位病人来了，挂号台（门控网络）先看病人的症状，然后只把病人送给最匹配的 2 位医生。绝大多数医生对这位病人是"沉默的"，不参与诊断。医院总接诊能力（参数量）是 8 倍，但每次接诊只花了 2 位医生的时间（计算量）。

这就是 MoE 的精髓：**参数量按专家数线性增长，但每次推理的计算量几乎不变**。

**核心直觉**：

- **Dense 模型**：每个 token 都经过全部参数（所有神经元都"开灯"）
- **MoE 模型**：每个 token 只激活 k 个专家（只有少数"灯"亮起），其余专家参数在本次推理中为零贡献
- 关键洞见：不同的输入（单词/语义片段）适合由不同专家处理——"专家自发学会分工"

## 📖 层次二：概念定义与基本原理

### 正式定义

MoE（Mixture of Experts）是一种**条件计算（Conditional Computation）**架构，由若干独立的子网络（专家，Expert）和一个门控网络（Gating Network）组成。给定输入 x，门控网络决定激活哪 k 个专家，最终输出是被激活专家输出的加权和：

```
y = Σ_{i ∈ Top-K} G(x)_i · E_i(x)
```

其中 G(x) 是稀疏门控向量，仅 k 个位置非零。

### 在 Transformer 中的位置

现代 MoE-LLM 将 **FFN 层替换为 MoE 层**，Attention 层保持不变：

```
Standard Transformer Block:
  x → LayerNorm → Multi-Head Attention → Add
  x → LayerNorm → FFN(Linear+Act+Linear) → Add

MoE Transformer Block:
  x → LayerNorm → Multi-Head Attention → Add
  x → LayerNorm → MoE(Gating + k×Expert_FFN) → Add
```

### 核心原理

| 原理 | 说明 |
|------|------|
| **稀疏激活** | 每个 token 只激活 k（通常 k=1 或 2）个专家，计算量与 k 成正比而非与专家总数 n 成正比 |
| **参数扩容** | 总参数量 ≈ 非MoE参数 + n × 单专家参数；激活参数 ≈ 非MoE参数 + k × 单专家参数 |
| **专家专化** | 训练收敛后，不同专家自发学习处理不同语义/语法特征（Shazeer 2017 Table 9 实证） |
| **负载均衡** | 若无约束，门控网络趋向将所有 token 路由到同一批"热门专家"，需用辅助损失或硬约束强制均衡 |

### 与相关概念的区别

| 概念 | 含义 | 与 MoE 的关系 |
|------|------|--------------|
| Dense Model | 所有参数对每个 token 都激活 | MoE 的对立架构 |
| Sparse Attention | 注意力只关注部分 token | 不同维度的稀疏；MoE 是 FFN 稀疏化 |
| LoRA | 低秩分解微调，减少训练参数 | 参数效率的另一路线；推理时参数全激活 |
| Ensemble | 多个独立模型投票 | 每个 token 激活所有模型；MoE 只激活 k 个 |
| MoE + LoRA | 对每个专家分别做 LoRA 微调 | 组合使用，已有 MoLoRA 等实践 |

## ⚙️ 层次三：技术细节

### 数学描述

**Noisy Top-K Gating**（Shazeer 2017 奠基方案）：

```
H(x)_i = (x · W_g)_i + ε · Softplus((x · W_noise)_i)    -- 加可学习噪声
G(x)   = Softmax(KeepTopK(H(x), k))                       -- Top-K 稀疏化

KeepTopK(v, k)_i = { v_i  if v_i ∈ top-k values of v
                    { -∞   otherwise
```

**负载均衡损失**（防止专家崩塌）：

```
Importance(X)_i = Σ_{x∈X} G(x)_i
L_importance = w_imp · CV(Importance(X))²         -- Importance Loss

P(x, i) = Φ((H(x)_i - kth_value) / Softplus((x·W_noise)_i))
Load(X)_i = Σ_{x∈X} P(x, i)
L_load = w_load · CV(Load(X))²                    -- Load Loss（可微分）
```

其中 CV 为变异系数（标准差/均值），两个损失均以各自系数（论文中 w=0.1）加入总 loss。

**Token Dropping**（Switch Transformer 硬约束）：

```
capacity_factor c：每个专家最多处理 ⌊c · tokens/experts⌋ 个 token
超出 capacity 的 token 被丢弃（residual connection 直通）
```

### 关键变体全景

| 方案 | 发布 | k 值 | 专家数 | 核心改进 |
|------|------|:---:|:------:|---------|
| Sparsely-Gated MoE | Shazeer 2017 | 4 | 2048 | 奠基：Noisy Top-K + 负载均衡双损失 |
| GShard | Google 2021 | 2 | 2048 | Transformer 适配；expert capacity 概念 |
| Switch Transformer | Google 2021 | 1 | 2048 | k=1 极简；capacity factor 硬约束；万亿参数 |
| GLaM | Google 2021 | 2 | 64 | GPT 风格；1.2T 参数；0-shot 超 GPT-3 |
| Mixtral 8×7B | Mistral 2023 | 2 | 8 | 开源；46.7B 总参/12.9B 激活；超 LLaMA-2 70B |
| DeepSeek-V2 | 2024 | 6/2 | 160+1 | 细粒度专家 + 共享专家；MLA 注意力 |
| DeepSeek-V3 | 2024 | 8 | 256+1 | 无辅助损失均衡（bias 调节）；671B 总参/37B 激活 |
| Qwen MoE | Alibaba 2024 | 4 | 60+4 | 共享专家 4 个；14.3B 总参/2.7B 激活 |

### 专家崩塌（Expert Collapse）问题

训练初期门控网络随机初始化，某些专家偶然比其他专家稍好 → 获得更多梯度 → 进一步变好 → 其余专家逐渐"废弃"。解决方案演进：

```
Noisy Top-K Gating（Shazeer 2017）  →  软约束，噪声打破对称性
         ↓
Capacity Factor + Token Dropping   →  硬约束，超额 token 直接跳过该专家
（Switch Transformer 2021）
         ↓
Expert Choice Routing（Zhou 2022）  →  反转路由：由专家主动选 token
         ↓
Auxiliary-loss-free（DeepSeek-V3）  →  动态 bias 调节，无辅助损失
```

## 💻 层次四：代码实现

核心 MoE 层的 PyTorch 精简实现（Noisy Top-K Gating，k=2）：

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


class Expert(nn.Module):
    """单个专家 = 标准 FFN"""
    def __init__(self, d_model: int, d_ff: int):
        super().__init__()
        self.fc1 = nn.Linear(d_model, d_ff)
        self.fc2 = nn.Linear(d_ff, d_model)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        return self.fc2(F.gelu(self.fc1(x)))


class NoisyTopKRouter(nn.Module):
    """Noisy Top-K 门控网络（Shazeer 2017）"""
    def __init__(self, d_model: int, n_experts: int):
        super().__init__()
        self.W_g     = nn.Linear(d_model, n_experts, bias=False)
        self.W_noise = nn.Linear(d_model, n_experts, bias=False)
        self.n_experts = n_experts

    def forward(self, x: torch.Tensor, k: int, training: bool):
        # x: (batch*seq, d_model)
        logits = self.W_g(x)                                   # (B, n_experts)

        if training:
            noise_scale = F.softplus(self.W_noise(x))
            noise = torch.randn_like(logits) * noise_scale     # 可学习噪声幅度
            logits = logits + noise

        # Top-K 稀疏化：非 top-k 置 -inf
        topk_vals, topk_idx = logits.topk(k, dim=-1)
        sparse_logits = torch.full_like(logits, float('-inf'))
        sparse_logits.scatter_(1, topk_idx, topk_vals)

        gates = F.softmax(sparse_logits, dim=-1)               # (B, n_experts)
        return gates, topk_idx


class SparseMoELayer(nn.Module):
    """稀疏 MoE 层，替换 Transformer 中的 FFN"""
    def __init__(self, d_model: int, d_ff: int, n_experts: int, k: int = 2):
        super().__init__()
        self.n_experts = n_experts
        self.k = k
        self.router  = NoisyTopKRouter(d_model, n_experts)
        self.experts = nn.ModuleList([Expert(d_model, d_ff) for _ in range(n_experts)])

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # x: (batch, seq, d_model)
        B, S, D = x.shape
        x_flat = x.view(B * S, D)                              # (B*S, D)

        gates, topk_idx = self.router(x_flat, self.k, self.training)

        # 逐 token 计算被激活专家的输出
        output = torch.zeros_like(x_flat)
        for i in range(self.n_experts):
            # 找出路由到专家 i 的 token
            token_mask = (topk_idx == i).any(dim=-1)           # (B*S,)
            if not token_mask.any():
                continue
            expert_input  = x_flat[token_mask]                 # (m, D)
            expert_output = self.experts[i](expert_input)      # (m, D)
            # 加权加回（gates 中该专家的权重）
            weight = gates[token_mask, i].unsqueeze(-1)        # (m, 1)
            output[token_mask] += weight * expert_output

        return output.view(B, S, D)

    def load_balance_loss(self, x: torch.Tensor) -> torch.Tensor:
        """Importance Loss：惩罚专家门控值分布不均（CV²）"""
        B, S, D = x.shape
        x_flat = x.view(B * S, D)
        gates, _ = self.router(x_flat, self.k, self.training)
        importance = gates.sum(dim=0)                          # (n_experts,)
        cv_sq = importance.var() / (importance.mean() ** 2 + 1e-8)
        return cv_sq
```

**关键代码解释**：

- `W_noise` 学习每个专家的"噪声幅度"，使各专家在训练初期有机会被选到，打破对称性
- `scatter_` 将非 top-k 置 -inf，Softmax 后这些位置自然为 0，无需额外掩码
- 内层循环"按专家聚合 token"在生产实现中会改用 `einsum` + `scatter_add` 以向量化；此处为逻辑清晰版本
- `load_balance_loss` 返回的 CV² 应乘以系数（如 1e-2）后加入总 loss

## 🔬 层次五：前沿进展与工程应用

### 在哪些模型/系统中用到

| 模型 | 专家配置 | 激活参数 | 特点 |
|------|---------|:--------:|------|
| Switch Transformer | 2048 专家，k=1 | ~1/2048 FFN | Google 万亿参数实验平台 |
| Mixtral 8×7B | 8 专家，k=2 | 12.9B / 46.7B | 开源 MoE 标杆；HuggingFace 可直接运行 |
| DeepSeek-V2 | 160 细粒度+1 共享，k=6 | 21B / 236B | 细粒度+共享专家双创新 |
| DeepSeek-V3 | 256+1，k=8 | 37B / 671B | 无辅助损失均衡；671B 开源 |
| Qwen1.5-MoE-A2.7B | 60+4 共享，k=4 | 2.7B / 14.3B | 推理成本仅 Qwen1.5-7B 的 25% |

**推断（未官方确认）**：GPT-4、Gemini Ultra 业内普遍认为采用 MoE 架构，官方未披露细节。

### 工程实践中的注意事项

**显存分布**：

- MoE 总参数量远大于激活参数量，专家参数必须分布到多卡（Expert Parallelism）
- Mixtral 8×7B 全精度约需 ~90GB 显存（8 × 2 × 7B 参数）；4-bit 量化可降至 ~25GB

**Token Routing 通信开销**：

- EP（Expert Parallelism）下 token 需跨卡路由，产生 All-to-All 通信
- DeepSeek-V3 用 "topK + limited routing"（每个 token 最多路由到 M 个节点）控制通信量

**负载不均对 throughput 的影响**：

- Capacity Factor < 1 时，热门专家会丢弃 token，导致精度下降
- Capacity Factor > 1 时，显存浪费；通常取 1.0-1.25 作为 trade-off

**微调陷阱**：

- MoE 模型微调时，若 batch 太小，部分专家可能整个 batch 都未被激活，梯度稀疏
- 建议使用专家粒度更细（如 DeepSeek 细粒度）或确保足够大的有效 batch size

### 与 Scaling Laws 的关系

Shazeer 2017 发现：**相同计算预算下，MoE 的参数扩展效益远优于 Dense 模型**。

具体数据（1B Word 语言建模，Shazeer 2017 Figure 2）：

- 固定 ops/timestep，4 个专家与 Dense 基线相当
- 256 个专家 PPL 低 6 点
- 4096 个专家继续改善，但边际递减

推论：MoE 是"挑战 Dense Scaling Laws"的一条路——在不提高推理计算量的前提下获得更大参数量带来的性能提升。DeepSeek-V3 的 671B/37B 配比正是这一逻辑的极致体现。

### 改进方向与开放问题

**1. 专家粒度**：DeepSeek-V2 将 FFN 切割成更小的"细粒度专家"（单个专家维度缩小，专家数变多），提高路由灵活性。

**2. 共享专家（Shared Experts）**：DeepSeek-V2/Qwen MoE 设置若干"全局激活"专家，捕获所有 token 都需要的通用知识，稀疏专家专注于特化知识。

**3. 无辅助损失均衡**：DeepSeek-V3 用动态 bias 调节各专家被选中频率，取消辅助损失以避免梯度干扰主任务。

**4. Expert Choice Routing**（Zhou et al. 2022）：反转路由方向——由专家主动"选取"最匹配的 token，自然实现完美负载均衡；代价是不同 token 的激活专家数可能不同，批推理实现复杂。

**5. 长上下文 MoE**：专家路由在超长序列（128K+）下的 token dropping 行为研究不足。

**6. 多模态 MoE**：V-MoE（Riquelme et al. 2021）将 MoE 引入 ViT，图像 token 的专家利用率分布与文本差异显著，崩塌问题更难处理。

## 历史叙事：MoE 从边缘到主流

```
[Jacobs 1991] Adaptive Mixtures of Local Experts
    — 最初提出 MoE，EM 训练，小规模图像任务

[Jordan & Jacobs 1994] Hierarchical MoE
    — 层次化门控，理论奠基

[Bengio 2013/2015] 条件计算理论
    — 提出概念框架，但无法在 GPU 大规模落地

[Shazeer et al. 2017] Sparsely-Gated MoE ← 现代 MoE 奠基
    — GPU 规模验证；Noisy Top-K；负载均衡双损失；137B 参数

[GShard 2021] MoE × Transformer，多语言 600B

[Switch Transformer 2021] k=1；万亿参数；capacity factor

[GLaM 2021] 1.2T；GPT 风格 0-shot 超越 GPT-3

[Mixtral 8×7B 2023] 开源 MoE SOTA；HuggingFace 可运行

[DeepSeek-V2 2024] 细粒度+共享专家；236B/21B

[DeepSeek-V3 2024] 671B/37B；无辅助损失均衡；开源
    — MoE 成为开源前沿模型主流架构
```

**现状**：截至 2026 年，MoE 已从"学术有趣但工程复杂"演变为前沿开源 LLM 的标准架构之一，与 Dense 模型并驾齐驱。DeepSeek-V3 的成功证明大规模 MoE 可以以显著更低的训练 FLOPs 达到 Dense 模型（如 Llama-3.1 405B）相当的性能。
