## 🔍 层次一：5岁小孩也能懂的类比

**类比故事**：想象你在考试，考卷有100道选择题，每道题都要你"回顾之前所有题目的选项"才能作答。如果每次作答都从第一题开始重新读一遍所有题，效率极低。但如果有一本"草稿本"，每道题做完就把题目和答案抄上去，之后需要时直接翻草稿本——这本草稿本就是 **KV Cache**。

**核心直觉**：大语言模型每生成一个新词，都需要"看一眼"之前所有词的信息（这就是 Attention 机制）。没有缓存时，每次都重新计算；有了 KV Cache，历史词的中间计算结果被保存下来，只需计算当前新词的部分。

---

## 📖 层次二：概念定义与基本原理

### 正式定义

KV Cache（Key-Value Cache）是在 Transformer 自回归推理过程中，将每一层 Multi-Head Attention 的 **Key（K）矩阵** 和 **Value（V）矩阵** 缓存到显存中，以避免在生成新 token 时对历史 token 进行重复计算的一种推理优化技术。

### 核心原理

**为什么可以缓存？**

在自回归生成时，模型按 token 逐个生成输出。设当前已生成 t 个 token，生成第 t+1 个时需要计算：

```
Attention(Q, K, V) = softmax(QK^T / √d_k) · V
```

其中：
- **Q**（Query）：仅由第 t+1 个 token 的表示生成，每步都是新的
- **K, V**：由前 t 个 token 的表示生成，**与上一步相比只新增了第 t 个 token 的部分**

因此：
- 历史 token 的 K, V 不需要重新计算，只需追加最新 token 的 K, V
- 这使得每步推理从 O(t²) 的计算量降至 O(t) 的增量计算

### 与相关概念的区别

| 概念 | 含义 | 与 KV Cache 的关系 |
|------|------|-------------------|
| Attention Cache | 有时与 KV Cache 同义 | 同一概念的不同叫法 |
| Prefix Caching | 缓存固定前缀（System Prompt）的 KV | KV Cache 的特殊应用场景 |
| PagedAttention | 将 KV Cache 分页管理（vLLM） | KV Cache 的内存管理方案 |
| FlashAttention | IO-aware 的 Attention 计算内核 | 与 KV Cache 正交，可同时使用 |

## ⚙️ 层次三：技术细节

### 数学描述

设序列长度为 T，隐藏维度为 d，头数为 h，每头维度为 d_k = d/h：

**无 KV Cache（每步从头计算）**：
```
# 生成第 t 个 token，序列长度已为 t
Q_t = X_t · W_Q          # shape: (1, d_k)
K_t = X_{1:t} · W_K      # shape: (t, d_k)  ← 重复计算！
V_t = X_{1:t} · W_V      # shape: (t, d_v)  ← 重复计算！
out = softmax(Q_t · K_t^T / √d_k) · V_t
```

计算 K, V 的总量：1 + 2 + ... + T = O(T²)

**有 KV Cache（增量计算）**：
```
# 缓存中已有 K_{1:t-1}, V_{1:t-1}
Q_t = x_t · W_Q           # shape: (1, d_k)
k_t = x_t · W_K           # shape: (1, d_k)  ← 只算新 token
v_t = x_t · W_V           # shape: (1, d_v)

K_t = concat(K_{cache}, k_t)   # 追加
V_t = concat(V_{cache}, v_t)   # 追加

out = softmax(Q_t · K_t^T / √d_k) · V_t

# 更新缓存
K_{cache} ← K_t
V_{cache} ← V_t
```

计算 K, V 的总量：T 次各 O(1) = O(T)，节省量级：O(T²) → O(T)

### 显存占用计算

单层 KV Cache 的显存：

```
KV_mem = 2 × seq_len × num_heads × head_dim × bytes_per_param

# 以 LLaMA-2 7B 为例：
# num_layers=32, num_heads=32, head_dim=128, seq_len=4096, fp16
KV_mem = 2 × 4096 × 32 × 128 × 2 bytes × 32 layers
       ≈ 2 GB
```

对于长上下文（128K token），KV Cache 可达 **数十 GB**，往往是推理显存的主要来源。

### 常见变体

**1. Multi-Query Attention (MQA)**（Shazeer, 2019）
- 所有 Q 头共享同一组 K、V
- KV Cache 显存减少 h 倍
- 代价：部分精度损失

**2. Grouped-Query Attention (GQA)**（Ainslie et al., 2023）
- 将 Q 头分组，每组共享一组 K、V
- MQA（1组）和 MHA（h组）之间的权衡
- LLaMA-2 70B、Mistral 7B 等主流模型采用

**3. Prefix Caching / Prompt Caching**
- 固定的 System Prompt 的 KV 可提前计算并缓存
- 多轮对话中节省首次生成时延
- OpenAI、Anthropic 均已支持

**4. PagedAttention**（vLLM, Kwon et al., 2023）
- 类比操作系统虚拟内存的分页机制
- 将 KV Cache 分成固定大小的"Block"
- 解决显存碎片化，提升吞吐量 5-10x

## 💻 层次四：代码实现

```python
import torch
import torch.nn as nn
import torch.nn.functional as F
import math

class MultiHeadAttentionWithKVCache(nn.Module):
    """带 KV Cache 的多头注意力（推理专用）"""

    def __init__(self, d_model: int, num_heads: int):
        super().__init__()
        self.d_model = d_model
        self.num_heads = num_heads
        self.d_k = d_model // num_heads

        self.W_q = nn.Linear(d_model, d_model, bias=False)
        self.W_k = nn.Linear(d_model, d_model, bias=False)
        self.W_v = nn.Linear(d_model, d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)

    def forward(
        self,
        x: torch.Tensor,           # shape: (batch, 1, d_model) — 每次只输入新 token
        past_key: torch.Tensor,    # shape: (batch, heads, past_len, d_k)
        past_val: torch.Tensor,    # shape: (batch, heads, past_len, d_v)
    ):
        B, seq_q, _ = x.shape

        # 1. 只计算新 token 的 Q, K, V
        q = self.W_q(x).view(B, seq_q, self.num_heads, self.d_k).transpose(1, 2)
        k = self.W_k(x).view(B, seq_q, self.num_heads, self.d_k).transpose(1, 2)
        v = self.W_v(x).view(B, seq_q, self.num_heads, self.d_k).transpose(1, 2)

        # 2. 拼接历史缓存（核心步骤）
        k_full = torch.cat([past_key, k], dim=2)   # (B, heads, past+1, d_k)
        v_full = torch.cat([past_val, v], dim=2)   # (B, heads, past+1, d_v)

        # 3. 标准 Attention 计算（Q 只有当前 token，K/V 包含全部历史）
        scale = math.sqrt(self.d_k)
        scores = torch.matmul(q, k_full.transpose(-2, -1)) / scale
        attn = F.softmax(scores, dim=-1)
        out = torch.matmul(attn, v_full)

        # 4. 输出投影
        out = out.transpose(1, 2).contiguous().view(B, seq_q, self.d_model)
        out = self.W_o(out)

        # 5. 返回输出 + 更新后的缓存（供下一步使用）
        return out, k_full, v_full


def autoregressive_generate(model, prompt_tokens, max_new_tokens=50):
    """演示 KV Cache 在自回归生成中的使用方式"""
    # Prefill 阶段：处理整个 prompt，获得初始 KV Cache
    past_key, past_val = None, None

    # 逐 token 生成（Decode 阶段）
    tokens = prompt_tokens
    for _ in range(max_new_tokens):
        # 只送入最后一个 token（使用缓存）
        x = tokens[:, -1:]

        logits, past_key, past_val = model(x, past_key, past_val)
        next_token = logits.argmax(dim=-1)
        tokens = torch.cat([tokens, next_token], dim=1)

    return tokens
```

**关键代码解释**：
- `torch.cat([past_key, k], dim=2)`：将新 token 的 K 追加到历史缓存，不重新计算历史
- `q` 只有 shape `(B, heads, 1, d_k)`：每步 decode 只计算当前 token 的 Query
- `k_full, v_full` 在每步后返回：调用方负责保存并传入下一步

## 🏗️ 层次五：在实际系统中的应用

### 主流推理框架中的实现

| 系统 | KV Cache 策略 | 特色 |
|------|-------------|------|
| **vLLM** | PagedAttention | 分页管理，最高吞吐量 |
| **TGI (HuggingFace)** | 连续批处理 + KV Cache | 生产级部署 |
| **TensorRT-LLM** | 静态 KV Cache 分配 | 延迟最低 |
| **llama.cpp** | 滚动窗口缓存 | 边缘设备友好 |
| **Ollama** | 基于 llama.cpp | 本地部署简化 |

### 主流模型的 GQA 配置

| 模型 | 注意力类型 | Q头数 | KV头数 | KV压缩比 |
|------|-----------|-------|--------|---------|
| LLaMA-2 7B | MHA | 32 | 32 | 1x |
| LLaMA-2 70B | GQA | 64 | 8 | 8x |
| Mistral 7B | GQA | 32 | 8 | 4x |
| GPT-3 175B | MHA | 96 | 96 | 1x |
| Gemma 7B | MQA | 16 | 1 | 16x |

### 工程实践中的注意事项

**陷阱1：显存随上下文线性增长**
- 长上下文（32K+）时 KV Cache 可能成为显存瓶颈
- 解决方案：量化（INT8/INT4 KV）、Window Attention、Streaming LLM

**陷阱2：批处理时 KV Cache 对齐问题**
- 不同序列长度的请求共享 GPU 时，padding 造成显存浪费
- vLLM 的 PagedAttention 通过动态分配解决此问题

**优化建议：Prefix Caching**
- 对于固定 System Prompt 场景，提前计算并缓存其 KV
- 重复请求（如 RAG 中的固定上下文）可节省 80%+ 的 TTFT（首 Token 时延）

**量化 KV Cache**
```python
# INT8 量化 KV Cache 示例（节省 50% 显存）
k_quantized = k.to(torch.int8)   # 量化存储
k_restored = k_quantized.float() # 使用时反量化
```

## 🔬 深度扩展：前沿进展

### 改进方向

**1. KV Cache 压缩**
- **H2O**（Heavy-Hitter Oracle）：识别并保留最重要的 KV 对，驱逐不重要的
- **SnapKV**（2024）：通过 Attention 权重识别关键 KV，压缩率 1000x 仅损失 0.5%
- **PyramidKV**：不同层使用不同压缩率（浅层保留更多）

**2. 长上下文 KV 管理**
- **Streaming LLM**（MIT, 2023）：保留 Attention Sink（前几个 token）+ 滑动窗口
- **LongCache**：分层 KV Cache，热数据在 GPU，冷数据卸载到 CPU/NVMe

**3. 跨请求 KV 复用**
- **Radix Attention**（vLLM v0.4+）：树状前缀共享，相同前缀的请求共用 KV
- **SGLang RadixCache**：为 multi-turn、RAG 场景优化

### 开放问题

- 超长上下文（1M token）下，KV Cache 的有效管理仍是挑战
- 如何在不损失精度的情况下将 KV Cache 压缩到 2-4 bit？
- 跨层 KV 共享是否可行（减少层数但保留表达力）？
